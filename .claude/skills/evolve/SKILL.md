# evolve — DRAFTプラグイン改善サイクル

## 目的

draft-pluginを使っていて感じた違和感・アイデアを、その場でヒアリングから実装・検証まで回し切り、プラグインと方法論を進化させる。

**思想: 課題をファイルに溜めない。** バックログやメモを残す代わりに、「こうできたらいいな」と思った瞬間にこのスキルを起動して改善を完了させる。見送った課題も記録しない——また思い出したら、もう一度evolveを回せばいい。

## 前提(入力)

- このスキルはDRAFTリポジトリ(draft-pluginのソースがある場所)でのみ使う。`draft-plugin/.claude-plugin/plugin.json` が存在しない場所で起動された場合は、DRAFTリポジトリで実行するよう案内して終了する
- 入力は「雑な違和感・アイデア」でよい。整理されている必要はない

## このリポジトリの変更対象マップ

| 対象 | 役割 |
|------|------|
| `README.md` | **方法論の原典**(6フェーズ+イテレーションの定義) |
| `draft-plugin/skills/*/SKILL.md` | 方法論の**実装**(8スキル: guide / concept / brainstorm / design-spec / playbook / prep / implement / feature) |
| `draft-plugin/skills/*/` 内のテンプレート | `templates/` 配下の**原本のコピー**(スキル実行時に参照される) |
| `templates/` | テンプレート原本 |
| `draft-plugin/.claude-plugin/plugin.json` | プラグインマニフェスト(バージョン管理) |
| `.claude-plugin/marketplace.json` | マーケットプレイス定義 |
| `draft-plugin/README.md` | プラグインの説明書(スキル一覧表・インストール手順) |

## 進め方

### 1. 雑なヒアリング

ユーザーの最初の一言は雑でよい。以下を軽く聞いて課題の輪郭を掴む(選択式・1テーマずつ。AskUserQuestion推奨。2〜3往復で十分、全部聞く必要はない):

- いつ・どのスキル(フェーズ)を使っていて感じたか
- 「困りごとの解消」か「新しく欲しいもの」か
- 理想の状態を一言でいうと

会話の流れで課題がすでに特定できている場合は、ヒアリングを省略して2に進む。

### 2. 影響範囲の特定

課題がどこに属するかを判定し、変更対象を列挙する:

| 課題の種類 | 変更対象 |
|-----------|---------|
| スキルの挙動・説明 | 該当する `SKILL.md` |
| 方法論そのもの | `README.md` と該当 `SKILL.md` の**両方**(真実の源が2つある。片方だけ直すと矛盾する) |
| テンプレートの内容 | `templates/` の原本と `skills/` 内コピーの**両方** |
| 新スキルの追加 | `skills/` に新ディレクトリ + `guide`(診断表・案内先) + `draft-plugin/README.md` の表 + 必要なら `README.md` |
| 配布・構成 | `plugin.json` / `marketplace.json` |

### 3. 方針の壁打ち

選択肢をトレードオフ付きで提示し、採用するか見送るかをユーザーと決める。

- **見送り・保留になったら、記録を残さずそのまま終了してよい。** バックログファイルは作らない
- 1回のevolveで扱う課題は1つに絞る。壁打ち中に別の課題が出てきたら「それは次のevolveで」と分ける

### 4. 実装

以下の規約を守って変更を適用する:

- SKILL.mdのフロントマターは `name` / `description` のみ。`name` はディレクトリ名と一致させる。`description` は「何をするか+いつ使うか」を日本語で(descriptionに `/draft: ` のようなコロン+スペースを含める場合はダブルクォートで囲む)
- 本文は日本語・命令形。構成: 目的 / 前提(入力) / 進め方 / 成果物 / 次のフェーズ / 注意
- スキル間の参照は `/draft:<name>` 形式で、参照先が実在すること
- 成果物パスの受け渡し(あるスキルの出力ファイル名=次のスキルの入力)を壊さない
- README(原典)に影響する変更は、READMEも同時に更新する
- テンプレート変更は原本(`templates/`)とコピー(`skills/` 内)の両方に適用する
- `plugin.json` のversionを上げる: 文言修正・バグ修正=patch、スキルの追加・挙動変更=minor
- スキルの追加・削除・改名をしたら、`guide` の診断表・案内先一覧と `draft-plugin/README.md` のスキル表も更新する

### 5. 検証(必須)

このスキルのディレクトリにある検証スクリプトをリポジトリルートから実行する:

```bash
ruby .claude/skills/evolve/validate.rb
```

チェック内容: 全SKILL.mdのフロントマター(YAMLパース・キー規約・name一致)、マニフェストJSONの妥当性、`/draft:` 参照先スキルの実在、同梱テンプレートファイルの実在、原本とコピーの乖離警告。

**エラーが0になるまで完了を宣言しない。** 警告(原本とコピーの乖離)は、意図した変更なら理由を確認した上で許容してよい。

### 6. 反映と報告

1. 変更の要約を報告する(何をなぜ変えたか、versionをいくつに上げたか)
2. プラグインをインストール済みの場合の反映手順を案内する: `/plugin marketplace update draft-marketplace` → `/reload-plugins`(または再起動)
3. 変更したスキルの動作確認方法を提案する(例: 該当スキルを実際に呼んでみる)
4. コミットはユーザーが希望した場合のみ行う

## 成果物

- 更新されたSKILL.md / README / テンプレート / マニフェスト(検証スクリプトをパスした状態)
- バックログ・メモファイルは**作らない**

## 次のフェーズ

なし(このスキルは繰り返し使う)。改善したスキルを実際に使ってみて、また違和感があればevolveを回す。

## 注意

- **課題をファイルに溜めない。** 思った瞬間に回し切るのがこのスキルの存在意義。「あとでまとめてやる」への逆行を許さない
- **README(原典)とスキル(実装)の同期を最優先。** 片方だけの修正は次の改善時の事故のもと
- **検証スクリプトを飛ばさない。** 過去にYAMLパースエラーや存在しないスキルへの参照が実際に混入している
- **1回のevolveは1課題。** 大きくなったら分割して次のevolveに回す
