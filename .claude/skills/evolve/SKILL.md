# evolve — DRAFTプラグイン改善サイクル

## 目的

draft-pluginを使っていて感じた違和感・アイデアを、その場でヒアリングから実装・検証まで回し切り、プラグインと方法論を進化させる。

**思想: 課題をファイルに溜めない。** バックログやメモを残す代わりに、「こうできたらいいな」と思った瞬間にこのスキルを起動して改善を完了させる。見送った課題も記録しない——また思い出したら、もう一度evolveを回せばいい。

## 前提(入力)

- このスキルはDRAFTリポジトリ(draft-pluginのソースがある場所)でのみ使う。`draft-plugin/.claude-plugin/plugin.json` が存在しない場所で起動された場合は、DRAFTリポジトリで実行するよう案内して終了する
- 入力は「雑な違和感・アイデア」でよい。整理されている必要はない

## このリポジトリの変更対象マップ

このリポジトリ自身にもDRAFTの3層を適用する: **README(原典)とtemplates/原本=ソース、スキルとテンプレートコピー=派生(READMEの実装)、変更の経緯=git履歴。**
真実の源は常にソース側の1箇所(方法論のspecはREADME、テンプレートはtemplates/原本)。食い違いは「どちらが正か」の問題ではなく「派生が古い」というバグであり、常にソースに合わせて直す。

| 対象 | 層 | 役割 |
|------|----|------|
| `README.md` | ソース | **方法論の原典**(6フェーズ+イテレーションの定義。specの正) |
| `draft-plugin/skills/*/SKILL.md` | 派生 | 方法論の**実装**(READMEに追随する。1スキル=1ディレクトリ。一覧は `draft-plugin/README.md` のスキル表) |
| `draft-plugin/skills/*/` 内のテンプレート | 派生 | `templates/` 配下の**原本のコピー**(スキル実行時に参照される配布物) |
| `templates/` | ソース | テンプレート原本 |
| `draft-plugin/.claude-plugin/plugin.json` | — | プラグインマニフェスト(バージョン管理) |
| `.claude-plugin/marketplace.json` | — | マーケットプレイス定義 |
| `draft-plugin/README.md` | — | プラグインの説明書(スキル一覧表・インストール手順) |

## 進め方

### 1. 雑なヒアリング

ユーザーの最初の一言は雑でよい。以下を軽く聞いて課題の輪郭を掴む(選択式・1テーマずつ。AskUserQuestion推奨。2〜3往復で十分、全部聞く必要はない):

- いつ・どのスキル(フェーズ)を使っていて感じたか
- 「困りごとの解消」か「新しく欲しいもの」か
- 理想の状態を一言でいうと

会話の流れで課題がすでに特定できている場合は、ヒアリングを省略して2に進む。

### 2. 影響範囲の特定

課題がどこに属するかを判定し、変更対象を列挙する:

| 課題の種類 | 変更対象 |
|-----------|---------|
| スキルの実行の仕方だけ(対話手順・ツール使用・エラー処理) | 該当する `SKILL.md` のみ。READMEには波及しない |
| 方法論そのもの(フェーズ定義・原則・成果物) | `README.md`(ソース)を**先に**直し、該当 `SKILL.md` を実装として追随させる。追随までが1サイクル |
| テンプレートの内容 | `templates/` の原本を直し、`skills/` 内コピーへ反映する(コピーは配布用の派生) |
| 新スキルの追加 | `skills/` に新ディレクトリ + `guide`(診断表・案内先) + `draft-plugin/README.md` の表 + 必要なら `README.md` |
| 配布・構成 | `plugin.json` / `marketplace.json` |

### 3. 方針の壁打ち

選択肢をトレードオフ付きで提示し、採用するか見送るかをユーザーと決める。

- **見送り・保留になったら、記録を残さずそのまま終了してよい。** バックログファイルは作らない
- 1回のevolveで扱う課題は1つに絞る。壁打ち中に別の課題が出てきたら「それは次のevolveで」と分ける

### 4. 実装

以下の規約を守って変更を適用する:

- SKILL.mdのフロントマターは `name` / `description` のみ。`name` はディレクトリ名と一致させる。`description` は「何をするか+いつ使うか」を日本語で(descriptionに `/draft: ` のようなコロン+スペースを含める場合はダブルクォートで囲む)
- 本文は日本語・命令形。構成: 目的 / 前提(入力) / 進め方 / 成果物 / 次のフェーズ / 注意
- スキル間の参照は `/draft:<name>` 形式で、参照先が実在すること
- 成果物パスの受け渡し(あるスキルの出力ファイル名=次のスキルの入力)を壊さない
- 方法論に関わる変更はREADME(ソース)を先に直し、スキル・テンプレコピーを追随させる(影響範囲マップの方向に従う)
- テンプレート変更は原本(`templates/`)を直し、コピー(`skills/` 内)へ反映する
- `plugin.json` のversionを上げる: 文言修正・バグ修正=patch、スキルの追加・挙動変更=minor
- スキルの追加・削除・改名をしたら、`guide` の診断表・案内先一覧と `draft-plugin/README.md` のスキル表も更新する

### 5. 検証(必須)

このスキルのディレクトリにある検証スクリプトをリポジトリルートから実行する:

```bash
ruby .claude/skills/evolve/validate.rb
```

チェック内容: 全SKILL.mdのフロントマター(YAMLパース・キー規約・name一致)、マニフェストJSONの妥当性、`/draft:` 参照先スキルの実在、同梱テンプレートファイルの実在、原本とコピーの乖離警告、README(原典)の `templates/` 参照の実在、スキル一覧の同期(guideの案内先・`draft-plugin/README.md` の表と構成ツリー)。

**エラーが0になるまで完了を宣言しない。** 警告(原本とコピーの乖離)は「派生が古い」というバグなので、原本に合わせてコピーを更新してから完了する(意図した恒久的乖離は存在しない)。

### 6. 反映と報告

1. 変更の要約を報告する(何をなぜ変えたか、versionをいくつに上げたか)
2. プラグインをインストール済みの場合の反映手順を案内する: `/plugin marketplace update draft-marketplace` → `/reload-plugins`(または再起動)
3. 変更したスキルの動作確認方法を提案する(例: 該当スキルを実際に呼んでみる)
4. コミットはユーザーが希望した場合のみ行う

## もう1つのモード: ルールGC(削る)

evolveは課題を「足す」だけでなく「削る」ためにも使う。方法論の決まり事は放っておくと増える一方で、官僚化する。肥大を感じた瞬間に回す(定期スケジュールやGC待ちリストは作らない — 思想は通常モードと同じ)。

決まり事には2種類ある:

- **恒久**: 意図・ファクト・検証に関わるルール(誰のため、スコープ、事実の所有権、受け入れ基準、独立レビュー)。モデルが賢くなっても残る
- **足場**: 今のAIの制約を補うルール(チャット分割、変換ステップ、手順の明文化、チェックリスト)。モデルの進化で理由が失効する

GCの対象は足場だけ。手順:

1. 足場ルールを1つ選ぶ。理由が書かれていないルールは、まず理由を特定して書く(理由の不明なルールは削除も維持も判断できない)
2. そのルールの理由が現行モデルでもまだ生きているかを挑戦する(ルール無しで該当フェーズを試す、モデルの実挙動と理由の記述を突き合わせる)
3. 理由が失効していたら削除する。READMEから消し、派生(スキル・テンプレコピー)を追随させる
4. 以降(検証・versionのbump・反映と報告・コミット)は通常モードの4〜6に従う。コミットメッセージに何を・なぜ削ったかを残す(経緯はgitが担う)

削るか迷ったら、ルールの理由を現行の観察に合わせて書き直す — それ自体を今回の改善として完了してよい(理由が正確になっていれば、いつでも判断し直せる)。

## 成果物

- 更新されたSKILL.md / README / テンプレート / マニフェスト(検証スクリプトをパスした状態)
- バックログ・メモファイルは**作らない**

## 次のフェーズ

なし(このスキルは繰り返し使う)。改善したスキルを実際に使ってみて、また違和感があればevolveを回す。

## 注意

- **課題をファイルに溜めない。** 思った瞬間に回し切るのがこのスキルの存在意義。「あとでまとめてやる」への逆行を許さない
- **ソースを直したら、派生(スキル・テンプレコピー)を追随させるまでが1サイクル。** 方法論のspecの正はREADME、テンプレートの正はtemplates/原本。スキル側だけを直して方法論を変えない(specと実装が乖離する)
- **検証スクリプトを飛ばさない。** 過去にYAMLパースエラーや存在しないスキルへの参照が実際に混入している
- **1回のevolveは1課題。** 大きくなったら分割して次のevolveに回す
