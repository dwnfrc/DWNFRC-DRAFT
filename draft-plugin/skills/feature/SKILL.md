# イテレーション: 機能追加サイクル

## 目的

Phase 5(初期実装)完了後のプロジェクトに、機能を1つ追加するサイクルを回す。
フル構築より軽量で、1機能あたり「壁打ち → Feature Design Doc(FDD)→ 実装プロンプト生成 → 実装 → ドキュメント更新」で完結させる。

大原則: **後工程に渡すのは常にファイル。口頭の合意ではなくMarkdownに書き出す。**
FDDは「変更の経緯(なぜ変えたか)」、System Design Docは「現在の状態(今どうなっているか)」という役割分担を常に維持する。

## 前提(入力)

作業を始める前に、必ず以下をReadする(壁打ちを省略する小規模変更でも省略不可):

1. `docs/design-spec.md`(設計仕様書)
2. `docs/02-01_system-design-doc.md`(System Design Doc)

この2つは既存設計との整合性を判断するために必須。読まずに設計すると既存設計と矛盾する提案をしてしまう。
ファイルが見つからない場合はユーザーに場所を確認し、存在しない場合は先に /draft:design-spec(Phase 2)や /draft:playbook(Phase 3)でのフェーズ実行を案内する(フェーズを飛ばさない)。

加えて、追加したい機能の概要(2〜3行で十分)をユーザーから聞く。まだ曖昧でもよい — 壁打ちで具体化する。

## 進め方

### Step 1: 変更規模の判定

機能概要と既存設計を照らし合わせ、まず変更規模を判定してユーザーに提示・合意する:

| 変更規模 | 判断基準 | 進め方 |
|---------|---------|--------|
| **小** | 既存画面内の修正、APIパラメータ追加程度 | 壁打ちなしでFDDを直接書いてOK → Step 3へ |
| **中** | 新規画面1〜2枚追加、スキーマ変更あり | 壁打ちでFDDを作成 → 1セッションで実装 |
| **大** | 新ロール追加、コアフロー変更 | 壁打ちでFDD作成 → 設計仕様書も更新 → 複数セッションで実装 |

規模判定に迷う場合は大きい方に倒す。小規模でもFDDの作成自体は省略しない(Step 3参照)。

### Step 2: 壁打ち(中・大規模のみ)

対話で機能の設計を詰める。以下の観点を必ず検討する:

- **既存の画面・APIへの影響範囲** — どの画面・エンドポイントに手が入るか
- **DBスキーマの変更が必要か** — 必要なら、マイグレーション計画(既存データの扱い含む)をこの段階で詰める
- **新規画面・エンドポイントが必要か** — 必要なら名称・目的・入出力の概要まで

対話の進め方:

- **選択式で聞き返す。** AskUserQuestionツールを使い、選択肢を提示して選んでもらう。オープンな質問を投げっぱなしにしない
- **1テーマずつ進める。** 影響範囲 → スキーマ → 画面/API のように、複数の論点を一度に聞かない
- **完璧を求めず先に進める。** ユーザーが「それで良い」と言ったら深掘りをやめて次へ。後でFDDを直せばよい
- 既存設計と矛盾する要望が出たら、その場で指摘し、どちらを変えるか選択肢で確認する

**大規模の場合:** 壁打ちの結論として設計仕様書(`docs/design-spec.md`)本体の更新も必要になる。壁打ち内で「設計仕様書のどのセクションをどう変えるか」まで合意し、FDD作成と合わせて設計仕様書も更新する。

### Step 3: Feature Design Doc の作成

壁打ちの合意(小規模なら自分の設計判断)をFDDとしてファイル化する。

1. このスキルのディレクトリにある `feature-design-doc.template.md` をReadし、その構成に従う:
   背景と目的 / Goal・Non-Goal / 影響範囲 / 実装アプローチ / 詳細設計(API変更・UI変更・データマイグレーション) / テスト方針 / ロールアウト計画
2. ファイル名のタイムスタンプはBashで取得する。丸めたり推測したりしない:
   ```bash
   date +%Y%m%d-%H%M
   ```
3. `docs/features/YYYYMMDD-HHMM_{機能名}.md` として保存する(例: `docs/features/20260709-1446_notification.md`)。ディレクトリがなければ作成する
4. 原則「1ユーザーストーリー = 1 FDD」。複数の独立した機能が混ざっていたら分割を提案する
5. スキーマ変更がある場合、詳細設計のデータマイグレーション欄に手順(スキーマ差分・既存データの移行・ロールバック方法)を具体的に書く
6. 保存後、ユーザーにFDDの内容を要約提示し、承認を得てから次へ進む

### Step 4: 実装プロンプトの生成

**FDDをそのまま実装指示にしない。** FDDには背景や検討経緯が含まれ、実装指示としては冗長。「何を作るか」(FDD)と「どう指示するか」(実装プロンプト)を分離する。

以下の原則でFDDを実装プロンプトに変換する:

- **実装の依存関係順に並べる**: スキーマ変更 → API → UI
- **各ステップに明確なゴール**を「〜が動く状態」の形で定義する(例:「マイグレーションが通り、seedデータが投入できる状態」)
- **参照すべきdocsファイルを明示する**: 最低限 `docs/features/YYYYMMDD-HHMM_{機能名}.md` と、関連する `docs/02-01_system-design-doc.md` のセクション
- **「まだやらないこと」(Non-Goal)を明示**してスコープクリープを防ぐ
- **末尾にドキュメント更新指示を必ず含める**(Step 6の内容。別作業にすると忘れるため、実装プロンプトに組み込む)
- 大規模の場合はセッションを分割できる粒度(1ステップ = 1セッション)に区切る

その後の実行方法をユーザーに確認する:

- **小〜中規模**: ユーザーが同一セッションでの実装を望むなら、生成したプロンプトに従ってこのままStep 5の実装に進んでよい
- **大規模**: 新しいセッションでの実行を推奨する。生成したプロンプトをコードブロックでそのままコピペできる形で提示し(Markdownの崩れに注意)、必要ならファイルにも保存する。ただしユーザーが同一セッションで続けることを選んだ場合は従う

### Step 5: 実装

実装を始める前に、/draft:branch で作業場所を決める(小規模はmain継続が普通。中〜大規模や影響範囲が広い変更はブランチ作成を検討し、ユーザーに選んでもらう)。

実装プロンプトのステップ順に実装する。各ステップの完了時に、定義したゴール(「〜が動く状態」)を実際に動作確認してから次のステップへ進む。動作確認せずに積み上げない。

- スキーマ変更があれば、マイグレーションが通ることを最初に確認する
- 生成したコードはセキュリティ観点(認証ガード、バリデーション)を必ずレビューする
- 実装中にFDDと異なる判断をした場合は、その場でFDDに追記して差分を残す

### Step 6: ドキュメント更新(実装完了・動作確認後)

実装が完了し動作確認が済んだら、ドキュメントを現在の実装に合わせて更新する:

- **System Design Doc**(`docs/02-01_system-design-doc.md`):
  - API設計セクション(追加/変更したエンドポイント)
  - データモデルセクション(スキーマ変更があった場合)
  - 画面一覧(画面追加があった場合)
- 上記3点に限らず、更新が必要なドキュメント(dev-setup、deployment、runbook等)は全て更新する

**なぜ実装後に更新するのか:**

- System Design Docは次のイテレーションの壁打ちで入力として使う。常に「現在の状態」を反映していないと、次の設計が古い情報を元に行われてしまう
- 実装中に方針が変わることがあるため、確定した状態を反映するのが正しいタイミング
- FDDは「変更の経緯」を残すドキュメント、System Design Docは「現在の状態」を表すドキュメント、という役割分担。FDDは書き換えずに蓄積し、System Design Docは常に最新化する

### Step 7: コミット

/draft:commit を実行し、リポジトリの流儀に合ったメッセージ・粒度でコミットする(実装とドキュメント更新の変更をまとめて扱う。コミット・プッシュはユーザーの合意があってから行う)。ブランチで作業していた場合は、マージ/PRの進め方をブランチ戦略(`docs/03_dev-setup.md`)に従って確認する。

## 成果物

| 成果物 | 場所 |
|--------|------|
| Feature Design Doc | `docs/features/YYYYMMDD-HHMM_{機能名}.md` |
| 実装コード | リポジトリ内 |
| 更新されたSystem Design Doc ほか関連ドキュメント | `docs/` 配下 |
| (大規模のみ)更新された設計仕様書 | `docs/design-spec.md` |

`docs/features/` にはタイムスタンプ付きFDDが時系列に蓄積され、プロジェクトの変更履歴になる。

## 次のフェーズ

このサイクルを機能ごとに繰り返す。次の機能を追加するときは、再度 /draft:feature を実行する(新しいセッションでの開始を推奨。ただしユーザーが同一セッションで続ける場合は従う)。
更新済みのSystem Design Docが次の壁打ちの入力になるため、Step 6を完了させてから次のサイクルに入ること。

## 注意

- **壁打ちの前に必ず設計仕様書とSystem Design DocをReadする。** 既存設計との整合性を判断するために必要。読まずに始めると既存設計と矛盾する設計が出てくる
- **小さな変更でもFDDを残す。** タイムスタンプ付きで蓄積すれば変更履歴になる
- **FDDをそのまま実装指示にしない。** 設計書と実装指示は役割が違う。実装プロンプトを生成するステップを必ず挟む
- **スキーマ変更がある場合はマイグレーション計画を先に詰める。** 実装後に「マイグレーションが通らない」となると手戻りが大きい
- **ドキュメント更新は実装フローに組み込む。** 別作業にすると忘れる。実装プロンプトの末尾に更新指示を含めておく
- **実装後にSystem Design Docを更新しないまま終わらない。** 次の壁打ちで古い情報を元に設計してしまう
- **動作確認せずに次のステップへ進まない。** エラーが積み上がると修正が困難になる
- タイムスタンプは必ず `date +%Y%m%d-%H%M` の実行結果を使う。丸めない・推測しない
