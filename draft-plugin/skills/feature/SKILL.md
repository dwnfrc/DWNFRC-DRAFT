# イテレーション: 機能追加サイクル

## 目的

Phase 5(初期実装)完了後のプロジェクトに、機能を1つ追加するサイクルの**前半(設計)**を担う。
サイクル全体は「壁打ち → Feature Design Doc(FDD)→ 実装プロンプト生成 → 実装 → ドキュメント更新 → コミット」で、このスキルはFDDの作成・承認までを行い、以降は /draft:feature-implement が引き継ぐ。設計と実装の受け渡しはFDDファイルで行う。

大原則: **後工程に渡すのは常にファイル。口頭の合意ではなくMarkdownに書き出す。**
FDDは「変更の経緯(なぜ変えたか)」、System Design Docは「現在の状態(今どうなっているか)」という役割分担を常に維持する。

## 前提(入力)

作業を始める前に、必ず以下をReadする(壁打ちを省略する小規模変更でも省略不可):

1. `docs/design-spec.md`(設計仕様書)
2. `docs/02-01_system-design-doc.md`(System Design Doc)

この2つは既存設計との整合性を判断するために必須。読まずに設計すると既存設計と矛盾する提案をしてしまう。
ファイルが見つからない場合はユーザーに場所を確認し、存在しない場合は先に /draft:design-spec(Phase 2)や /draft:playbook(Phase 3)でのフェーズ実行を案内する(フェーズを飛ばさない)。

加えて、追加したい機能の概要(2〜3行で十分)をユーザーから聞く。まだ曖昧でもよい — 壁打ちで具体化する。

## 進め方

### Step 1: 変更規模の判定

機能概要と既存設計を照らし合わせ、まず変更規模を判定してユーザーに提示・合意する:

| 変更規模 | 判断基準 | 進め方 |
|---------|---------|--------|
| **小** | 既存画面内の修正、APIパラメータ追加程度 | 壁打ちなしでFDDを直接書いてOK → Step 3へ |
| **中** | 新規画面1〜2枚追加、スキーマ変更あり | 壁打ちでFDDを作成 |
| **大** | 新ロール追加、コアフロー変更 | 壁打ちでFDD作成 → 設計仕様書も更新 |

規模判定に迷う場合は大きい方に倒す。小規模でもFDDの作成自体は省略しない(Step 3参照)。

### Step 2: 壁打ち(中・大規模のみ)

対話で機能の設計を詰める。以下の観点を必ず検討する:

- **既存の画面・APIへの影響範囲** — どの画面・エンドポイントに手が入るか
- **DBスキーマの変更が必要か** — 必要なら、マイグレーション計画(既存データの扱い含む)をこの段階で詰める
- **新規画面・エンドポイントが必要か** — 必要なら名称・目的・入出力の概要まで

対話の進め方:

- **選択式で聞き返す。** AskUserQuestionツールを使い、選択肢を提示して選んでもらう。オープンな質問を投げっぱなしにしない
- **1テーマずつ進める。** 影響範囲 → スキーマ → 画面/API のように、複数の論点を一度に聞かない
- **完璧を求めず先に進める。** ユーザーが「それで良い」と言ったら深掘りをやめて次へ。後でFDDを直せばよい
- 既存設計と矛盾する要望が出たら、その場で指摘し、どちらを変えるか選択肢で確認する

**大規模の場合:** 壁打ちの結論として設計仕様書(`docs/design-spec.md`)本体の更新も必要になる。壁打ち内で「設計仕様書のどのセクションをどう変えるか」まで合意し、FDD作成と合わせて設計仕様書も更新する。

### Step 3: Feature Design Doc の作成

壁打ちの合意(小規模なら自分の設計判断)をFDDとしてファイル化する。

1. このスキルのディレクトリにある `feature-design-doc.template.md` をReadし、その構成に従う:
   背景と目的 / Goal・Non-Goal / 影響範囲 / 実装アプローチ / 詳細設計(API変更・UI変更・データマイグレーション) / テスト方針 / ロールアウト計画
2. ファイル名のタイムスタンプはBashで取得する。丸めたり推測したりしない:
   ```bash
   date +%Y%m%d-%H%M
   ```
3. `docs/features/YYYYMMDD-HHMM_{機能名}.md` として保存する(例: `docs/features/20260709-1446_notification.md`)。ディレクトリがなければ作成する
4. 原則「1ユーザーストーリー = 1 FDD」。複数の独立した機能が混ざっていたら分割を提案する
5. スキーマ変更がある場合、詳細設計のデータマイグレーション欄に手順(スキーマ差分・既存データの移行・ロールバック方法)を具体的に書く
6. **中・大規模の場合、承認を求める前に独立レビューを行う。** サブエージェント(Agentツール)で、壁打ちの経緯を知らないレビュアーを起動し、FDDと `docs/02-01_system-design-doc.md` だけを渡して「既存設計との整合」「マイグレーション計画の具体性」「Goal/Non-Goalの明確さ」を検証させる(会話の経緯は渡さない — 先入観なく矛盾を見つけるため)。指摘は自分で反映する。小規模ではスキップしてよい
7. 保存後、ユーザーにFDDの内容を要約提示し、承認を得てから次へ進む

### Step 4: 実装への引き継ぎ

FDDの承認後、実装の進め方をユーザーに確認する:

- **同一セッションで続ける**(小〜中規模で文脈が軽いとき): /draft:feature-implement を実行してそのまま実装に進む
- **新しいセッションで実行する**(大規模、またはこのセッションが長いとき。推奨): 新しいセッションで /draft:feature-implement を実行するよう案内する。受け渡しはFDDファイルなので、プロンプトのコピペや引き継ぎ説明は不要

いずれの場合も、実装セッションが対象を特定できるよう、FDDのパス(`docs/features/YYYYMMDD-HHMM_{機能名}.md`)を明示して締める。

## 成果物

| 成果物 | 場所 |
|--------|------|
| Feature Design Doc(承認済み) | `docs/features/YYYYMMDD-HHMM_{機能名}.md` |
| (大規模のみ)更新された設計仕様書 | `docs/design-spec.md` |

`docs/features/` にはタイムスタンプ付きFDDが時系列に蓄積され、プロジェクトの変更履歴になる。
実装コードとSystem Design Docの更新は /draft:feature-implement の成果物。

## 次のフェーズ

/draft:feature-implement で対象FDDを実装する(Step 4で確認した進め方に従い、同一セッション続行または新しいセッションで実行)。
実装とドキュメント更新が完了したら、次の機能はまた /draft:feature で新しいサイクルを開始する。

## 注意

- **壁打ちの前に必ず設計仕様書とSystem Design DocをReadする。** 既存設計との整合性を判断するために必要。読まずに始めると既存設計と矛盾する設計が出てくる
- **小さな変更でもFDDを残す。** タイムスタンプ付きで蓄積すれば変更履歴になる
- **設計と実装を混ぜない。** FDDの承認前に実装を始めない。実装は /draft:feature-implement の担当
- **スキーマ変更がある場合はマイグレーション計画を先に詰める。** 実装後に「マイグレーションが通らない」となると手戻りが大きい
- タイムスタンプは必ず `date +%Y%m%d-%H%M` の実行結果を使う。丸めない・推測しない
