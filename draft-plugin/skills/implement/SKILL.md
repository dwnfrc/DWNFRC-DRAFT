# Phase 5: 実装(ステップ実行)

## 目的

Phase 4で生成したステップ別プロンプト集(`docs/claude-code-prompts.md`)のうち、**1ステップだけ**を実装し、動作確認済みの状態で完了させる。全ステップを順に消化していくことで、動くアプリケーションを完成させる。

DRAFTの原則「**1ステップ = 1チャット**」を厳守する(`docs/claude-code-prompts.md` が同一チャットでの実行を明記しているステップ群のみ例外)。1セッションで複数ステップを続けて実行すると、コンテキストが溢れて品質が低下する。

## 前提(入力)

以下がリポジトリに存在することを確認する:

- `CLAUDE.md`(リポジトリルート)— プロジェクトコンテキスト
- `docs/claude-code-prompts.md` — ステップ別プロンプト集
- `docs/` 配下のPhase 2〜3成果物(design-spec.md、02-01_system-design-doc.md など)

`docs/claude-code-prompts.md` が見つからない場合はリポジトリルートも探す。それでも無ければ、**このフェーズを実行してはいけない**。Phase 4(実装準備)が未完了である旨を伝え、先に `/draft:prep` でCLAUDE.mdとプロンプト集を生成するよう案内して終了する(`CLAUDE.md` が無い場合も同様に `/draft:prep` へ案内する)。フェーズを飛ばしてその場でアプリを作り始めることは絶対にしない。

## 進め方

### 1. ステップ一覧の把握

`docs/claude-code-prompts.md` をReadし、以下を整理する:

- 全ステップの一覧(番号・内容・ゴール)
- 各ステップの「動作確認可能なゴール」(「〜が動く状態」)
- 各ステップが参照を指示しているdocsファイル
- 各ステップの「まだやらないこと」(スコープ外)

### 2. 実行ステップの決定(ユーザーに確認)

プロジェクトの現状から**未完了の最初のステップ**を推定する。推定材料:

- リポジトリのファイル構成(package.json / スキーマ定義 / APIルート / 認証コード / UIコンポーネント / terraform / テストの有無)
- `git log --oneline` の履歴
- `docs/claude-code-prompts.md` 内に完了マーク(チェックボックス等)があればそれ

推定結果を踏まえ、**AskUserQuestionツールで選択式に確認する**:

- 第一候補: 推定した「次のステップ」(推定理由を添える)
- その他の選択肢: 前後のステップ、または「別のステップを指定する」

ユーザーが最初から「Step Nをやって」と指定している場合は確認を省略してよいが、そのステップより前に未完了のステップがありそうな場合は依存関係の観点から一言指摘する(最終判断はユーザーに従う)。

**例外:** `docs/claude-code-prompts.md` がチャット分割の目安として同一チャットでの実行を明記しているステップ群(例:「Step 1〜2は1チャットにまとめてもOK」)は、1セッションでまとめて実行してよい。それ以外は1ステップ = 1セッションを厳守する。

### 3. 実装(選ばれた1ステップのみ)

1. ステップのプロンプトが参照を指示しているdocsファイルを**先にRead**する。必要な情報だけ読み、docs全部を読み込まない
2. プロンプトの指示に従って実装する。技術スタック・設計方針は `CLAUDE.md` と `docs/02-01_system-design-doc.md` に従い、**このフェーズで技術スタックを変更しない**
3. 「まだやらないこと」に挙げられた項目には手を出さない(スコープクリープ防止)
4. **1ステップが大きすぎると感じたら分割する。** 進めてみてゴールまでが遠いと判断したら、ユーザーに分割を提案し、前半のみをこのセッションで完了させる。残りは `docs/claude-code-prompts.md` に新しいステップとして追記し、次のセッションに回す
5. **TODO/FIXMEコメントを残さない。** TODOを書きたくなったら、プロンプトか設計に曖昧さがあるサイン。先送りせず、その場でユーザーに確認して解消してから実装する(必要なら該当docsにも反映する)

**エラーが出たら場当たり的に直さない。** エラーメッセージを読み、根本原因(依存関係、設定、設計との不整合など)を特定し、原因と修正方針を提示してから修正する。設計に関わる修正(スキーマ変更、設計との不整合の解消など)はユーザーの合意を得てから行う。症状だけ抑える暫定パッチを重ねると、後のステップでエラーが積み上がって修正困難になる。

### 4. 独立レビュー(必須)

動作確認の前に、生成したコードをレビューする。サブエージェント(Agentツール)で、実装の経緯を知らないレビュアーを起動し、このステップのdiffと参照したdocsだけを渡して検証させる(実装中の会話や判断の経緯は渡さない — 書いた本人の思い込みを引き継がずに欠陥を見つけるため):

- 認証ガード: protected routesに認可チェックが漏れなく入っているか
- バリデーション: APIの入力検証が全エンドポイントにあるか
- シークレット: APIキーやパスワードがハードコードされていないか(.env参照になっているか)
- 残骸: TODO/FIXMEコメントが残っていないか。実ファイルを追加したディレクトリに `.gitkeep` が残っていないか(残っていたら削除する)
- docs整合: 実装が参照したdocsの記述から逸脱していないか。逸脱がある場合、それは合意済みの設計変更としてソースDocに反映されているか

指摘は重大度付きで返させ、修正は自分で行う。重大な指摘は修正後に再確認する。サブエージェントが使えない環境では自分でレビューする(その場合も上の観点を全て確認する)。

### 5. 動作確認(必須)

ステップの種類に応じて、以下の表の該当する確認を**必ず実行してから**完了を宣言する:

| ステップの種類 | 確認方法 |
|------|----------|
| 骨格(初期化・開発環境) | `make dev` で両方のサーバーが起動する |
| DBスキーマ + シード | `make db-push` + `make db-seed` + `make db-studio` でデータが見える |
| バックエンドAPI | `curl` で各エンドポイントが200を返す |
| 認証 | ブラウザでOAuthログインが通る |
| UI | ブラウザでコアフローが一通り動く |
| インフラ | `terraform plan` がエラーなく通る |
| テスト | `make test` が全パス |

(コマンド名はプロジェクトの `CLAUDE.md` / `docs/03_dev-setup.md` に定義されたものを優先する。Makefileが無い既存プロジェクトでは、そのプロジェクトの流儀のコマンドに従う。)

ブラウザ操作が必要な確認(OAuthログイン、コアフロー)は、自動で確認できない場合はサーバーを起動した上でユーザーに手動確認を依頼し、結果を聞いてから完了とする。

確認が失敗したら、根本原因を特定して修正し、再度確認する。**動作確認をパスするまで完了を宣言しない。**

### 6. 完了処理

1. **ソースDocの同期**: このステップの実装がdocsの記述から逸脱した場合(合意して行った設計変更・仕様調整)、変わった事実の所有ドキュメントを現在の実装に合わせて更新する(画面・UX・認証要否 → `docs/design-spec.md`、API・データモデル・ルーティング → `docs/02-01_system-design-doc.md`、コマンド・環境構築 → `docs/03_dev-setup.md`)。逸脱がなければ何もしない。更新は「現在の状態」の書き換えとして行い、更新履歴・編集ログは残さない
2. 実装内容の要約(作成・変更したファイル、動作確認の結果)をユーザーに報告する
3. `docs/claude-code-prompts.md` の該当ステップに完了マーク(例: `[x]` や `<!-- DONE: YYYY-MM-DD -->`)を記録する。次回セッションでの進捗推定に使うため、口頭報告だけでなくファイルに残す
4. /draft:commit でコミットする(メッセージと対象をユーザーに提示し、合意を得てから。勝手にコミットしない)

## 成果物

- 選択した1ステップの実装コード(**動作確認済み**)
- `docs/claude-code-prompts.md` 上の進捗記録

## 次のフェーズ

- **未完了のステップが残っている場合**: 次のステップは**新しいセッションで** `/draft:implement` を再度実行するよう案内する。CLAUDE.mdは新しいセッションでも自動で読まれ、作成したファイルはディスクに残っているため、引き継ぎ説明は不要である。ユーザーが同一セッションでの続行を強く希望する場合のみ従うが、コンテキスト溢れによる品質低下のリスクを一度は伝えること
- **全ステップ完了の場合**: フル構築は完了。以降の機能追加は `/draft:feature`(イテレーション: 壁打ち → Feature Design Doc → 実装)で行うと案内する

## 注意

- **全ステップを1つのセッションで実行しない。** コンテキスト溢れで品質が低下する。1ステップ完了ごとにセッションを切るのがDRAFTの設計
- **動作確認せずに次のステップへ進まない。** エラーが積み上がると原因の切り分けが困難になる
- **生成されたコードは必ず独立レビューにかける。** 書いた本人によるセルフレビューは思い込みを引き継ぐ。特にセキュリティ(認証ガード、バリデーション)。AIが生成したコードの認可漏れは典型的な事故ポイント
- **1ステップが大きすぎると感じたら分割する。** 完璧な粒度は事前にはわからない。途中で気づいたら遠慮なく提案する
- **技術スタックを実装フェーズで変更しない。** ドキュメント全体の書き直しになる。変更が本当に必要なら、いったん停止してPhase 3のドキュメントから直すようユーザーと合意する
- **場当たり的なエラー修正をしない。** 根本原因を特定し、原因と修正方針を提示してから直す。設計に関わる修正はユーザーと合意してから
- **TODOコメントで先送りしない。** 書きたくなったら設計の曖昧さを解消してから実装する。`.gitkeep` の残骸も含め、セルフレビューで掃除する
- **後工程に渡すのは常にファイル。** 進捗・決定事項はセッション内の合意で終わらせず、`docs/claude-code-prompts.md` 等に書き残す
