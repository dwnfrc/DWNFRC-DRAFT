# commit — 変更のコミット

## 目的

今ある変更を、適切な粒度と、リポジトリの既存の流儀に合ったメッセージでコミットする。

## 前提(入力)

- gitリポジトリで、コミットすべき変更があること
- 規約の情報源(あれば読む): `CLAUDE.md` のCode Style(Conventional Commits等)、`docs/03_dev-setup.md`(コミット/PRのルール)

## 進め方

### 1. 流儀の把握

1. **`git log` で直近のコミットメッセージを全文確認する(`--oneline` にしない)。** サブジェクトだけでは、ボディの書き方(言語・箇条書きの流儀・プレフィックスの使い方)がわからない
2. `CLAUDE.md` / `docs/03_dev-setup.md` にコミット規約が定義されていればそちらを優先する

### 2. 変更の把握と粒度の決定

1. `git status` と `git diff`(staged / unstaged の両方)で変更内容を把握する
2. ステージ済みの変更があればそれをコミット対象とする。無ければ、変更内容から論理的なまとまりを提案してステージする
3. **無関係な変更が混ざっていたら分割コミットを提案する。** 1コミット = 1つの意図
4. 対象にゴミ(.DS_Store等)やシークレット(.env の中身、APIキー、パスワード)が混ざっていないか確認する。ゴミは除外し、シークレットは除外した上で警告する

### 3. メッセージ生成とコミット

1. Step 1で把握した流儀に合わせてメッセージを作る(言語・時制・プレフィックス・ボディの詳しさ)
2. **署名やトレーラー(Co-Authored-By等)を勝手に付けない。** リポジトリの規約が明示的に要求している場合のみ付ける
3. メッセージと対象ファイル一覧をユーザーに提示し、合意を得てからコミットする
4. pushはしない(ユーザーが希望した場合のみ)

## 成果物

- 適切な粒度・流儀に合ったメッセージのコミット

## 次のフェーズ

- 実装の続きがある場合: 新しいセッションで /draft:implement(次のステップ)
- 機能追加サイクルが完了した場合: 次の機能は /draft:feature で新しいサイクルを開始

## 注意

- **git log は全文で読む。** onelineはメッセージスタイルの情報源として不十分
- **1コミット = 1つの意図。** 「ついで」の変更は分ける
- **push・履歴の書き換え(--amend、rebase)はpush済みのコミットに対して行わない。** 必要な場合はユーザーと合意してから
- コミットは必ずユーザーへの提示と合意の後で行う(勝手にコミットしない)
