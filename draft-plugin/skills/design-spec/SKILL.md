# Phase 2: 設計仕様書

## 目的

Phase 1（壁打ち）で合意した内容を「設計仕様書」として書き出す。
ゴールは、開発者（ユーザー自身を含む）が読んでそのまま画面が作れるレベルの具体性。
この仕様書はPhase 3（開発ドキュメント）以降のすべての工程が参照する情報源になる。

**大原則: 後工程に渡すのは常にファイル。口頭（チャット内）の合意ではなくMarkdownに書き出す。**

## 前提（入力）

1. `docs/brainstorm-notes.md`（Phase 1の壁打ちメモ）と `docs/concept.md`（Phase 0のコンセプト）をReadする
   - 同一セッションでPhase 1の壁打ち直後なら、その文脈をそのまま使ってよい。ただしその場合もファイルは必ずReadし、チャット内の記憶とファイルの内容がズレていないか確認する
2. `docs/brainstorm-notes.md` が存在しない場合はフェーズを飛ばさない。壁打ちが未実施であることをユーザーに伝え、先に `/draft:brainstorm` を実行するよう案内する（設計仕様書なしでPhase 3に進むと、API設計とUI設計が噛み合わなくなる）
3. このスキルのディレクトリにある `design-spec.template.md` と `screen_flow.template.mermaid` をReadする。これが成果物の構成の基準になる

## 進め方

### Step 1: 材料の確認

入力ファイルを読み、以下がPhase 1で合意済みかを確認する:

- 画面一覧（名称 + 目的 + 簡易説明）
- 画面遷移の全体像
- ロール定義
- コアフロー（最も重要なユーザー体験の流れ）

欠けている項目があれば、仕様書を書き始める前にAskUserQuestionで確認して埋める。
このとき壁打ちの流儀に従う: **選択式で聞き返す、1テーマずつ、完璧を求めず先に進める。**

### Step 2: 設計仕様書のドラフト作成

`design-spec.template.md` の9章構成に従って設計仕様書を書く:

1. プロダクト概要（コンセプト、方針、UI設計原則）
2. ユーザーロールとフロー
3. 画面一覧
4. レイアウト設計システム
5. 画面遷移（別紙 `screen_flow.mermaid` への参照）
6. コア画面詳細仕様
7. データモデル
8. デモデータ仕様
9. 今後の検討事項

このとき、以下の6項目は必ず含める（READMEのPhase 2プロンプトテンプレートに対応）:

1. プロダクト概要（コンセプト、方針（対応言語含む）、UI設計原則）
2. ユーザーロールとフロー
3. 画面一覧（全画面の名称・目的・レイアウト・認証要否）
4. コア画面の詳細仕様（各ステップで何が表示され、何が操作できるか。エラー・空・ローディング状態も含む）
5. データモデル（ER図 + 各テーブルのカラム定義）
6. 画面遷移図（Mermaid形式で**別ファイル**）

執筆時の指針:

- **仕様書は長くていい。** 網羅性より簡潔さを優先しない。Phase 3以降でAIが参照するための情報源なので、判断根拠や具体的な表示内容まで書き込む
- 画面一覧には全画面について「名称・目的・レイアウトパターン・認証要否」を表で記載する
- レイアウト設計システムは「画面の目的別にレイアウトパターンを定義 → 全画面にマッピング」の順で体系化する（例: 左右分割＝参照と操作を同時に行う画面、中央集中＝1つの対象に注目させる画面、カードグリッド＝比較・選択する画面）
- コア画面の詳細仕様にはASCIIレイアウト図を含め、各ステップ（状態）ごとに「何が表示され、何が操作できるか」を書く
- コア画面はエラー・空（データ0件）・ローディングの各状態で何が表示されるかも仕様化する（正常系だけの仕様書だと、実装時に状態設計が抜ける）
- プロダクト方針には対応言語を明記する（i18nの要否はここで決める。不要なら「日本語のみ」と書く。後付けは高くつく）
- データモデルは論理設計レベルでよい（Phase 3でORMスキーマコードに変換する前提）が、エンティティ・関係性・カラム定義はここで確定させる

### Step 3: コア画面の深掘り

コアフローに関わる画面や、仕様が曖昧な画面は、書きっぱなしにせずユーザーと対話して詰める。
AskUserQuestionを使い、画面ごとに以下の5観点で確認する:

- ユーザーが画面を開いたとき何が見えるか
- 操作可能なアクションは何か
- 各アクション後の状態遷移は何か
- エラー時・データが空のとき何が見えるか
- レイアウト（左右分割？中央集中？グリッド？）

レイアウトが画面ごとにバラバラになりそうな場合は、
「画面の目的別に最適なレイアウトパターンを定義し、全画面にマッピングする」体系化を提案して4章を固める。

深掘りも壁打ちの流儀で行う: 選択肢を提示して選んでもらう、1画面（1テーマ）ずつ、ユーザーが「それで良い」と言ったら先に進む。全画面を深掘りする必要はなく、コア画面と曖昧な画面に絞る。

### Step 4: 画面遷移図の作成

`screen_flow.template.mermaid` の形式に従い、画面遷移図をMermaidの `graph TD` で書く。

- 共通画面（LP、認証）→ ロール別の画面群、という構造でロールごとにセクションを分ける
- 設計仕様書の「3. 画面一覧」に載っている全画面が図に登場することを確認する
- **仕様書本文には埋め込まず、必ず別ファイルにする**

### Step 5: 書き出しと独立レビュー、最終確認

1. `docs/design-spec.md` と `docs/screen_flow.mermaid` に書き出す（`docs/` がなければ作成する）
2. **独立レビューを行う。** サブエージェント（Agentツール）で、作成の経緯を知らないレビュアーを起動し、書き出した2ファイルだけを渡して検証させる（壁打ちの会話や判断理由は渡さない — 先入観なく矛盾を見つけるため）:
   - 内部整合: 画面一覧・画面遷移図・データモデル・コア画面詳細仕様が互いに矛盾していないか（図に登場しない画面、仕様のないテーブル参照など）
   - 抜け: コア画面のエラー・空・ローディング状態、対応言語（i18n方針）が明記されているか
   指摘は重大度付きで返させ、修正は自分で行う。重大な指摘は修正後に再確認する。サブエージェントが使えない環境では、新しいセッションに成果物だけを渡してレビューする
3. 書き出した内容の要点（画面数、ロール数、テーブル数、コア画面）をユーザーに提示し、特に**データモデル**について最終確認を取る
4. 修正指示があればファイルを更新する

## 成果物

| ファイル | 内容 |
|----------|------|
| `docs/design-spec.md` | プロダクト設計の全体像（9章構成） |
| `docs/screen_flow.mermaid` | 画面遷移図（Mermaid形式、別ファイル） |

## 次のフェーズ

設計仕様書が固まったら、Phase 3（開発ドキュメント）へ進む。

- `/draft:playbook` を案内する
- 新しいセッションでの開始を推奨する（コンテキストを軽く保つため）。ただしユーザーが同一セッションで続けたい場合はそのまま続行してよい。成果物はすべて `docs/` のファイルとして残っているので、どちらでも品質は変わらない

## 注意

- **仕様書は「長くていい」。** Phase 3以降でAIが参照するための情報源。短くまとめようとして情報を削らない
- **データモデルはこの段階で固める。** 後から変えるとAPI設計から全部やり直しになる。曖昧なまま先に進めず、Step 5でユーザーの確認を必ず取る
- **Mermaid図は別ファイルにする。** 仕様書本文に埋め込むと後工程のAIが図を読み飛ばすことがある
- **フェーズを飛ばさない。** 壁打ち（Phase 1）の合意なしに仕様書を書くと方向性がブレる。逆に、設計仕様書なしでPhase 3（開発ドキュメント）を作るとAPI設計とUI設計が噛み合わない
- 技術スタックの選定はこのフェーズの仕事ではない（Phase 3で確定させる）。ユーザーが技術の話を始めたら、要望をメモとして「9. 今後の検討事項」に残しつつ、画面とデータの設計に集中させる
- 深掘り対話では完璧を求めない。ユーザーが決めきれない項目は妥当なデフォルトを提案して先に進み、「9. 今後の検討事項」に記録する
