# イテレーション: FDDの実装

## 目的

/draft:feature で作成・承認されたFeature Design Doc(FDD)を動くコードに落とす、イテレーション後半を担う。
「実装プロンプト生成 → 実装 → ドキュメント更新 → 独立レビュー → コミット」で1機能を完結させる。

大原則: **後工程に渡すのは常にファイル。** 設計フェーズからの受け渡しはFDDファイルであり、実装プロンプトはこのセッションでFDDから組み立てる。設計セッションで生成した文面が手元に残っていなくても、FDDがあれば再開できる。

## 前提(入力)

1. 対象のFDD(`docs/features/YYYYMMDD-HHMM_{機能名}.md`)。ユーザーの指定が無ければ、`docs/features/` を確認して未実装とみられるFDD(System Design Docに反映されていない・対応する実装が見当たらないもの)を候補として提示し、選んでもらう
2. `docs/02-01_system-design-doc.md`(System Design Doc)。FDDとの整合確認に必須

FDDが存在しない場合は、先に /draft:feature(設計: 壁打ち→FDD作成)を案内して終了する。設計を飛ばしてその場で実装を始めない。

## 進め方

### Step 1: FDDと現在の設計の突き合わせ

対象FDDと `docs/02-01_system-design-doc.md` をReadする。FDD作成後にSystem Design Docが更新されていて設計に影響する食い違いがある場合は、ユーザーに確認し、FDDを直してから先へ進む。

### Step 2: 実装プロンプトの生成

**FDDをそのまま実装指示にしない。** FDDには背景や検討経緯が含まれ、実装指示としては冗長。「何を作るか」(FDD)と「どう指示するか」(実装プロンプト)を分離する。

以下の原則でFDDを実装プロンプトに変換する:

- **実装の依存関係順に並べる**: スキーマ変更 → API → UI
- **各ステップに明確なゴール**を「〜が動く状態」の形で定義する(例:「マイグレーションが通り、seedデータが投入できる状態」)
- **参照すべきdocsファイルを明示する**: 最低限 対象FDDと、関連する `docs/02-01_system-design-doc.md` のセクション
- **「まだやらないこと」(Non-Goal)を明示**してスコープクリープを防ぐ
- **末尾にドキュメント更新指示を必ず含める**(Step 4の内容。別作業にすると忘れるため、実装プロンプトに組み込む)
- 大規模の場合は1ステップ = 1セッションで区切れる粒度に分割する

### Step 3: 実装

実装を始める前に、/draft:branch で作業場所を決める(小規模はmain継続が普通。中〜大規模や影響範囲が広い変更はブランチ作成を検討し、ユーザーに選んでもらう)。

実装プロンプトのステップ順に実装する。各ステップの完了時に、定義したゴール(「〜が動く状態」)を実際に動作確認してから次のステップへ進む。動作確認せずに積み上げない。

- スキーマ変更があれば、マイグレーションが通ることを最初に確認する
- 生成したコードはセキュリティ観点(認証ガード、バリデーション)を必ずレビューする
- **TODO/FIXMEコメントを残さない。** FDDが明確ならTODOは残らないはず。書きたくなったらFDDに曖昧さがあるサインなので、その場でユーザーに確認してFDDを更新してから実装する
- 実装中にFDDと異なる判断をした場合は、その場でFDDに追記して差分を残す
- 大規模でセッションを跨ぐ場合は、FDD末尾に進捗(どのステップまで完了したか)を追記してから切り上げ、次のセッションで再度 /draft:feature-implement を実行して続きから再開する

### Step 4: ドキュメント更新(実装完了・動作確認後)

実装が完了し動作確認が済んだら、ドキュメントを現在の実装に合わせて更新する:

- **System Design Doc**(`docs/02-01_system-design-doc.md`):
  - API設計セクション(追加/変更したエンドポイント)
  - データモデルセクション(スキーマ変更があった場合)
  - 画面一覧(画面追加があった場合)
- 上記3点に限らず、更新が必要なドキュメント(dev-setup、deployment、runbook等)は全て更新する
- docs/ に新しいファイルを追加した場合は `docs/README.md` の索引にも追記する(個別のFDDは索引に載せない。`features/` ディレクトリの案内で足りる)
- **更新は「現在の状態」の書き換えとして行う。** 更新履歴・編集ログ・「FDD◯◯により追加」のような出自の注記をドキュメントに残さない。変更の経緯はFDDとgit履歴が担う
- ADRの追加・更新はこの禁止の対象外。イテレーションで設計判断が変わった場合は、「現在の設計のなぜ」の最新化としてADRを追加・更新する

**なぜ実装後に更新するのか:**

- System Design Docは次のイテレーションの壁打ちで入力として使う。常に「現在の状態」を反映していないと、次の設計が古い情報を元に行われてしまう
- 実装中に方針が変わることがあるため、確定した状態を反映するのが正しいタイミング
- FDDは「変更の経緯」を残すドキュメント、System Design Docは「現在の状態」を表すドキュメント、という役割分担。FDDは書き換えずに蓄積し、System Design Docは常に最新化する

### Step 5: 独立レビュー(コミット前)

サブエージェント(Agentツール)で、実装の経緯を知らないレビュアーを起動し、実装のdiff・対象FDD・更新後の関連ドキュメントだけを渡して検証させる(実装中の会話や判断の経緯は渡さない — 先入観なく欠陥を見つけるため):

- 実装: FDDのGoal通りに動くか。認証ガード・バリデーションの漏れ、TODO/FIXMEコメント・不要な `.gitkeep` などの残骸
- ドキュメント: System Design Docほかの更新が実装の現在の状態と一致しているか。更新履歴・編集ログ・FDD参照の注記が混入していないか

指摘は重大度付きで返させ、修正は自分で行う。重大な指摘は修正後に再確認する。サブエージェントが使えない環境では、新しいセッションに成果物だけを渡してレビューする。

### Step 6: コミット

/draft:commit を実行し、リポジトリの流儀に合ったメッセージ・粒度でコミットする(実装とドキュメント更新の変更をまとめて扱う。コミット・プッシュはユーザーの合意があってから行う)。ブランチで作業していた場合は、マージ/PRの進め方をブランチ戦略(`docs/03_dev-setup.md`)に従って確認する。本番リリースが必要な場合は、同ドキュメントのリリースフロー(環境ごとのバージョン宣言ファイルを更新するpromotion PR)に従って進める。

## 成果物

| 成果物 | 場所 |
|--------|------|
| 実装コード(動作確認済み) | リポジトリ内 |
| 更新されたSystem Design Doc ほか関連ドキュメント | `docs/` 配下 |
| (実装中に判断が変わった場合)追記されたFDD | `docs/features/` |

## 次のフェーズ

次の機能を追加するときは /draft:feature で新しいサイクル(設計)を開始する(新しいセッションでの開始を推奨)。
更新済みのSystem Design Docが次の壁打ちの入力になるため、Step 4〜6(ドキュメント更新・独立レビュー・コミット)を完了させてから次のサイクルに入ること。

## 注意

- **FDDをそのまま実装指示にしない。** 設計書と実装指示は役割が違う。実装プロンプトを生成するステップを必ず挟む
- **設計を飛ばして実装しない。** FDDが無ければ /draft:feature に戻す
- **動作確認せずに次のステップへ進まない。** エラーが積み上がると修正が困難になる
- **場当たり的なエラー修正をしない。** 根本原因を特定し、原因と修正方針を提示してから直す
- **実装後にSystem Design Docを更新しないまま終わらない。** 次の壁打ちで古い情報を元に設計してしまう
- **ドキュメント更新は実装フローに組み込む。** 別作業にすると忘れる。実装プロンプトの末尾に更新指示を含めておく
