# Phase 3: 開発ドキュメント(PROJECT-PLAYBOOK)

## 目的

Phase 2の設計仕様書を「開発着手できる」レベルの技術ドキュメント一式に変換する。
PROJECT-PLAYBOOKテンプレートに従った6ファイルを docs/ に生成し、技術スタックをこのフェーズで確定させる。あわせて docs/ の入口となる `docs/README.md`(ドキュメントインデックス)を生成する。

| ドキュメント | 内容 |
|-------------|------|
| `01_prd.md` | 概要/背景と目的、ターゲットユーザーと課題、主要機能(ユーザーストーリー)、成功指標(KPI)、スコープ外 |
| `02-01_system-design-doc.md` | Goal/Non-Goal、アーキテクチャ概要、技術選定(ADR)、画面一覧、API設計、データモデル、セキュリティ・パフォーマンス、テスト戦略、モニタリング・ログ |
| `02-02_feature-design-doc.md` | 機能追加イテレーション用テンプレート(空のまま残す) |
| `03_dev-setup.md` | 環境構築手順、必要なツール・アカウント、ローカル開発サーバー起動、テスト実行、ブランチ戦略・リリースフロー、トラブルシューティング |
| `04_deployment-procedure.md` | 各環境へのデプロイ手順、リリース前チェックリスト、ロールバック手順、デプロイ後確認、緊急時連絡先 |
| `05_operation-runbook.md` | モニタリング・ログ設計、監視ポイントとアラート、よくある障害と対処法、ログ確認方法、定期メンテナンス |

## 前提(入力)

1. `docs/design-spec.md` と `docs/screen_flow.mermaid` をReadする。
   - **どちらかが存在しない場合は作業を開始しない。** 「設計仕様書なしで開発ドキュメントを作るとAPI設計とUI設計が噛み合わない」ため、まず `/draft:design-spec` (Phase 2) を実行するようユーザーに案内して終了する。
2. 新しいセッションでの実行を推奨する(コンテキストを設計仕様書に集中させるため)。ただしユーザーが同一セッションで続ける場合はそのまま従う。

## 進め方

### Step 1: 設計仕様書の理解

`docs/design-spec.md` と `docs/screen_flow.mermaid` を熟読し、以下を把握する:

- ユーザーロールとコアフロー
- 画面一覧と画面遷移
- データモデル(エンティティと関係性)

不明瞭な点が6ドキュメントの品質に影響する場合のみ、ユーザーに簡潔に確認する。細部の再議論はしない(Phase 2の役割)。

### Step 2: 技術スタックの確定

**技術スタックはこのフェーズで確定させる。** 後続フェーズで変えるとドキュメント全体の書き直しになる。

まずユーザーに技術スタックが決まっているか確認する。AskUserQuestionツールを使い、選択式で聞く。確認する項目:

```
- フロントエンド: [Next.js / React / etc.]
- バックエンド: [NestJS / Express / etc.]
- DB: [PostgreSQL / MySQL / etc.]
- 認証: [NextAuth / Firebase Auth / etc.]
- ホスティング: [Cloud Run / Vercel / etc.]
- IaC: [Terraform / なし]
```

**未定の場合**は一緒に決める。進め方:

1. まず方向性を聞く(例:「用途はポートフォリオ?事業?」「フロントはReact系?」「バックエンドの言語はTypeScript?」「クラウドはAWS?GCP?こだわりなし?」)。AskUserQuestionで選択式にし、1テーマずつ聞く。
2. 方向性に応じて、このスキルのディレクトリにある参考資料をReadして選択肢の提示に活用する:
   - `references/tech-stacks-aws.md` — AWS系スタックの選択肢と推奨
   - `references/tech-stacks-gcp.md` — GCP系スタックの選択肢と推奨
3. 用途・規模・ユーザーの経験に合わせて2〜3案を比較表(特徴・適している場合・コスト感)で提示し、推奨案を明示して選んでもらう。
4. 決定した項目ごとに「なぜそれを選んだか」「捨てた選択肢とのトレードオフ」をメモしておく(Step 3のADRの材料になる)。

完璧を求めず先に進める。迷ったら推奨案で仮確定し、確定したスタック一覧をユーザーに提示して最終合意を取る。

### Step 3: 6ドキュメントの生成

このスキルのディレクトリにある templates/ 配下の6テンプレートをReadする:

- `templates/01_prd.md`
- `templates/02-01_system-design-doc.md`
- `templates/02-02_feature-design-doc.md`
- `templates/03_dev-setup.md`
- `templates/04_deployment-procedure.md`
- `templates/05_operation-runbook.md`

テンプレートの構成(見出し・表の形式)に従い、設計仕様書と確定した技術スタックの内容で埋めて、ユーザーのプロジェクトの docs/ に同名で書き出す。

各ドキュメントの執筆指針:

**01_prd.md**
- 設計仕様書のコンセプト・ロール定義から、ターゲットユーザーと課題、ユーザーストーリーを書く
- ユーザーストーリーはロール別に整理し、優先度をつける
- 「スコープ外」を必ず書く(スコープクリープ防止)

**02-01_system-design-doc.md** — 最重要ドキュメント。手を抜かない:
- **ADR(Architectural Decision Records)は必ず書く。** Step 2の技術選定ごとに「決定・理由・トレードオフ」を記録する。「なぜこの技術を選んだか」がポートフォリオの差別化になる
- **API設計はリクエスト/レスポンスのボディまで書く。** エンドポイント一覧だけで済ませない。実装フェーズで「API設計に従って」と指示できる粒度にする
- **データモデルはER図に加えて、実際のORMコード(Prisma/Drizzle等、選定したORM)でスキーマ定義を書く。** そのままコピペで使える状態にする
- 画面一覧は設計仕様書から転記し、ルート・レイアウト・認証要否を確定させる

**02-02_feature-design-doc.md**
- **テンプレートの内容をそのまま docs/ にコピーする。埋めない。** Phase 5完了後の機能追加イテレーション(/draft:feature)で、Feature Design Doc作成の雛形として使う

**03_dev-setup.md**
- 確定したスタックに合わせて、ツール一覧・環境構築手順・コマンド一覧・ブランチ戦略・リリースフローを書く
- ブランチ戦略はテンプレートの標準(GitHub Flow + 常にsquash + 環境ごとのバージョン宣言ファイルによる環境プロモーション)をベースに、プロジェクトの環境構成(本番のみ / staging+本番)に合わせて調整する
- テンプレート内のコマンド例(GCP + Terraform + pnpm前提の箇所など)は、実際の構成に合わせて必ず書き換える
- 「環境構築30分以内」を目安に、コピペで進められる手順にする

**04_deployment-procedure.md**
- 確定したホスティング・IaC・CI/CDに合わせて書き換える(テンプレートの例はGCP + Terraform + GitHub Actions構成)
- リリーストリガー(`deploy/{環境}/version` の更新)とロールバック(promotion PRのrevert)を実際のCI/CDに合わせて具体化する
- リリース前チェックリスト、ロールバック手順、デプロイ後確認は省略しない

**05_operation-runbook.md**
- 確定したスタックで実際に起こりうる障害と対処法に書き換える(テンプレートの例はCloud Run/Cloud SQL前提)
- ログ確認コマンドは選定したホスティングの実際のCLIコマンドで書く

### Step 4: docs/README.md と PRテンプレートの生成

docs/ の入口として `docs/README.md` を生成する。テンプレートは無い。以下の構成で短く書く(1ページ以内):

- プロジェクト名と一行説明
- 読み順ガイド(初めて読む人はどの順で読むか。例: design-spec → 01_prd → 02-01 → 03_dev-setup)
- ドキュメント一覧表(ファイルへのリンク + 内容の一行説明)。Phase 2の成果物(`design-spec.md`、`screen_flow.mermaid`)も含める
- `features/` ディレクトリの案内(機能追加のFDDが時系列で蓄積される場所であること)

**実在するファイルだけを載せる。** 個別のFDDファイルは索引に載せない(イテレーションのたびに索引のメンテが必要になるため、`features/` ディレクトリとして案内する)。

あわせて `.github/PULL_REQUEST_TEMPLATE.md` を生成する(ディレクトリがなければ作成)。テンプレートは無い。以下の構成で短く書く:

- 概要(何を・なぜ)
- 関連ドキュメント(FDDや設計ドキュメントへのリンク欄)
- 変更内容
- 動作確認(実施した確認の記載欄)
- チェックリスト(セルフレビュー済み / 関連ドキュメント更新済み / スキーマ変更がある場合マイグレーション確認済み)
- 末尾にコメントで、promotion PR(リリース)の場合に書くこと(対象バージョン・stagingでの検証結果・ロールバック手順)を案内する

生成後、docs/README.md を含む7ファイルが docs/ に、`.github/PULL_REQUEST_TEMPLATE.md` がリポジトリに揃っていることを確認し、ユーザーに一覧と要点(特に技術スタックとADR)を報告してレビューを促す。修正依頼があればファイルを直接編集する。

## 次のフェーズ

6ドキュメントの生成とユーザーの確認が終わったら、Phase 4(実装準備)として `/draft:prep` を案内する。
CLAUDE.md とステップ別実装プロンプト集を作成するフェーズであり、新しいセッションで開始することを推奨する(ユーザーが同一セッションで続ける場合は従う)。

## 注意

- **技術選定はこのフェーズで確定させる。** Phase 4以降で変えると手戻りが大きい(アンチパターン:「技術スタックを実装フェーズで変更」→ ドキュメント全体の書き直し)
- **設計仕様書なしで進めない。** API設計とUI設計が噛み合わなくなる。無ければ必ず /draft:design-spec に戻す
- **ADRは必ず書く。** 「なぜこの技術を選んだか」がポートフォリオの差別化になる
- **API設計はリクエスト/レスポンスのボディまで書く。** バックエンド実装時のプロンプトで「API設計に従って」と指示できるようにする
- **DBスキーマは実際のコード(Prisma/Drizzle等)で書く。** そのままコピペで使える状態にする
- **02-02_feature-design-doc.md を埋めてしまわない。** これは将来のイテレーション用テンプレートであり、空のまま docs/ に残す
- **docs/README.md には実在するファイルだけを載せる。** 個別のFDDは載せず、`features/` ディレクトリとして案内する
- データモデルの大変更が必要だと判明した場合は、このフェーズで吸収せず Phase 2(design-spec.md)に戻って修正してから再開する。設計仕様書と開発ドキュメントの不整合を残さない
