# Phase 0: コンセプト作成

## 目的

「何を作りたいか」を1ページで言語化し、`docs/concept.md` として書き出す。
完璧なコンセプトを作ることが目的ではない。**メモ書きレベルで十分**。ここで作るファイルが次のPhase 1（壁打ち）の入力になる。

DRAFTの大原則を守ること: **後工程に渡すのは常にファイル。口頭の合意ではなくMarkdownに書き出す。**

## 前提（入力）

- ユーザーの頭の中にあるアイデア（口頭・断片で構わない）
- すでにメモや文章がリポジトリ内や手元にあれば、それも入力になる（ユーザーに確認し、あればReadする）
- 既存の `docs/concept.md` があれば、それを更新するモードで進める（上書き前に既存内容をReadして踏まえる）

事前のドキュメントは一切不要。このフェーズがDRAFTの出発点。

## 進め方

### 1. テンプレートを把握する

まず、このスキルのディレクトリにある `concept.template.md` をReadして構成を把握する。セクションは以下の7つ:

1. `# コンセプト: {プロジェクト名}`
2. 一言でいうと
3. 誰のために
4. どんな課題を解決するか
5. どんな体験を提供するか
6. なぜ作るのか（ポートフォリオ？事業化？学習？趣味？）
7. 思いついているアイデア（箇条書きでOK）

### 2. ユーザーの状態を確認する

ユーザーにアイデアの状態を聞く。すでに文章・メモがあるかどうかで分岐する:

- **メモや文章がすでにある場合** → それを見せてもらう（貼り付け、またはファイルパスを教えてもらってRead）。内容をテンプレートの7セクションに構造化して割り振る。足りないセクションだけを対話で補う。
- **頭の中にしかない場合** → まず「どんなものを作りたいか、思いつくままに話してください。雑でOKです」と自由に語ってもらう。その内容をセクションに割り振ったうえで、埋まらなかったセクションを1つずつ聞いていく。

### 3. 足りないセクションを1テーマずつ聞き出す

対話のルール（READMEの壁打ち原則に準拠）:

- **1回の質問で1テーマだけ**扱う。複数セクションをまとめて聞かない
- **選択式で聞き返す**。AskUserQuestionツールを使い、ユーザーの発言から推測した候補を2〜4個提示して選んでもらう。自由記述を強要しない
  - 例:「なぜ作るのか」→ 選択肢: ポートフォリオ / 事業化したい / 技術の学習 / 趣味・自分用
  - 例:「誰のために」→ ユーザーの語った内容から具体的なペルソナ候補を作って提示する
- **完璧を求めない**。ユーザーが「わからない」「まだ決めてない」と言ったら、`（未定 — Phase 1で詰める）` と書いてそのまま先に進む。空欄で止まらない
- ユーザーの回答は、そのままではなく**一文に整えて**テンプレートに反映する。ただし勝手に膨らませない。ユーザーが言っていないことを断定的に書かない
- プロジェクト名が未定なら仮の名前を提案して選んでもらう（後で変えられることを伝える）

### 4. docs/concept.md として書き出す

全セクションが埋まったら（未定を含んでよい）、ドラフト全文をユーザーに見せて確認を取る。修正希望があれば反映する。

確認が取れたら:

1. カレントリポジトリに `docs/` ディレクトリが無ければ作成する
2. `docs/concept.md` としてWriteする。テンプレートの見出し構成をそのまま使い、HTMLコメント（`<!-- -->` の説明書き）は削除して実際の内容だけを残す

チャット上の合意で終わらせず、必ずファイルに書き出すこと。

## 成果物

- `docs/concept.md` — 1ページのコンセプトメモ

以下が含まれていると後工程（Phase 1以降）がスムーズになるので、書き出す前に軽くセルフチェックする:

- 誰のためのプロダクトか
- どんな体験を提供するか
- 自分がなぜ作りたいか（ポートフォリオ？事業？学習？）

含まれていなくても止めない。「未定」と明記されていれば次フェーズで磨ける。

## 次のフェーズ

`docs/concept.md` の書き出しが完了したら、ユーザーに次を案内する:

> コンセプトメモができました。次は **/draft:brainstorm**（Phase 1: 壁打ち）で、このコンセプトを「画面とフローが見える」レベルまで具体化します。新しいセッションで始めるのがおすすめです（このまま続けても構いません）。

新しいセッションの開始を推奨するが、ユーザーが同一セッションで続けたい場合はそれに従う。

## 注意

READMEのTipsとアンチパターンより、このフェーズで守ること:

- **構成が雑でも問題ない。** Phase 1の壁打ちで構造化・具体化される。このフェーズで作り込みすぎない
- **「コンセプトに自信がない」状態でも先に進んでいい。** 壁打ちで磨かれる。自信を持たせようと質問を重ねすぎない
- **アンチパターン: コンセプトなしでいきなり実装に入らない。** ユーザーが「もうコード書いて」と言っても、方向性がブレて手戻りが大量になるため、まず1ページのコンセプトを書き出してからPhase 1〜4を踏むよう促す（それでもユーザーが強く望むなら従うが、リスクは一度伝える）
- **フェーズを飛ばさない。** このスキルの責務はコンセプトメモの作成まで。画面一覧・データモデル・技術選定の議論を始めそうになったら「それはPhase 1以降で詰めましょう」とメモ（思いついているアイデア欄）に退避して進める
- **成果物は必ずファイル。** 対話で満足して書き出しを忘れない。`docs/concept.md` が存在して初めてPhase 0完了
