# branch — 作業ブランチの準備

## 目的

これから始める作業(実装ステップ、機能追加)に対して、プロジェクトのブランチ戦略に沿った作業場所を決める。

**「ブランチを切らずmainのまま進める」も正しい結果の1つ。** 特にフル構築中(Phase 5)や小規模な変更ではmain直で進めるのが普通で、ブランチは影響範囲が広い変更のための安全弁として使う。

## 前提(入力)

- gitリポジトリであること
- ブランチ戦略の定義元: `docs/03_dev-setup.md` の「ブランチ戦略」セクション(無いプロジェクトでも動く。その場合はgitの現状とヒアリングから判断する)

## 進め方

### 1. 現状と戦略の把握

1. `git status` / `git branch` / `git log --oneline -5` で現在のブランチ・未コミット変更・直近の履歴を確認する
2. `docs/03_dev-setup.md` の「ブランチ戦略」をReadする。`CLAUDE.md` に関連する記述があればそれも読む
3. これから始める作業の内容と規模を把握する(会話の流れで明らかなら聞き直さない)

### 2. 戦略が未定義の場合: ヒアリングして文書化する

`docs/03_dev-setup.md` にブランチ戦略の記載が無い場合、AskUserQuestionで軽くヒアリングする(1問で十分):

- **main直コミット** — 1人開発・構築初期はこれが普通
- **GitHub Flow** — feature/xxx を切って main へマージ
- **規模で使い分け** — 小さい変更はmain直、影響範囲が広い変更はブランチ

合意した戦略を `docs/03_dev-setup.md` のブランチ戦略セクションに追記する(次回からは聞かずに済む。使うほどドキュメントが育つ)。`docs/03_dev-setup.md` 自体が無いプロジェクトでは追記せず、その回の判断としてだけ使う。

### 3. main継続かブランチ作成かを提案する

戦略と作業規模から提案し、ユーザーに選んでもらう:

| 状況 | デフォルト提案 |
|------|--------------|
| フル構築中(Phase 5のステップ実行) | main継続 |
| イテレーション: 小規模(既存画面内の修正、パラメータ追加程度) | main継続 |
| イテレーション: 中〜大規模(スキーマ変更、新規画面、コアフロー変更) | ブランチ作成 |
| 戦略がブランチ運用を明示している | 戦略に従う |

main継続を選んだ場合は、その旨を伝えて終了する(何もしないことがこのスキルの正しい出力になりうる)。

### 4. ブランチの作成

1. ブランチ名は戦略の命名規則に従う。規則が無ければ `feature/{機能名}` を基本とし、イテレーション中でFDDがあるならFDDファイル名の機能名と揃える(例: `docs/features/20260716-1030_notification.md` → `feature/notification`)
2. 未コミットの変更がある場合、ブランチに持ち越すか、先に /draft:commit でコミットしてから切り替えるかを確認する
3. リモートがある場合、ベースブランチ(通常main)が最新かを確認してから作成・切替する

## 成果物

- 作業場所の決定(main継続、または作成・切替済みのブランチ)
- (戦略が未定義だった場合)ブランチ戦略が追記された `docs/03_dev-setup.md`

## 次のフェーズ

作業本体へ進む: /draft:implement(Phase 5)または /draft:feature(イテレーション)。作業が完了したら /draft:commit でコミットする。

## 注意

- **ブランチ戦略をこのスキルにハードコードしない。** 真実の源は `docs/03_dev-setup.md`。プロジェクトごとに戦略は違う
- **main継続を「サボり」扱いしない。** 1人開発の構築期はmain直が最も速い。ブランチを強要しない
- **push・リモートブランチの作成は勝手にしない。** ユーザーの指示があってから行う
- 戦略を文書化したら、追記内容がヒアリングの合意と一致しているか読み返して確認する
