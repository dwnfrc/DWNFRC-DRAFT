# Phase 1: 壁打ち

## 目的

Phase 0のコンセプトを、対話を通じて「画面とフローが見える」レベルまで具体化する。
アプリの絵姿(どんな画面があり、誰がどう使うか)をユーザーと一緒に固め、合意事項をファイルに残す。

## 前提(入力)

1. まず `docs/concept.md` をReadする。
2. 存在しない場合は壁打ちを始めず、ユーザーに次のように案内して終了する:
   - 「Phase 0のコンセプトがまだありません。先に `/draft:concept` でコンセプトを1ページにまとめてください。」
   - ただしユーザーが「コンセプトは口頭で説明する」と言った場合は、その説明を聞いてから壁打ちに入ってよい。その内容も最終的に成果物へ反映する。
3. `docs/concept.md` があれば内容を読み込み、冒頭で「コンセプトをこう理解した」と1〜3行で要約して認識を合わせてから壁打ちを開始する。

## 進め方

### 対話の基本ルール

- **1テーマずつ進める。** 複数テーマを一度に質問しない。
- **選択式で聞き返す。** AskUserQuestionツールを使い、2〜4個の具体的な選択肢を提示する。ユーザーは直感で選んでよく、後で変えられることを伝える。
- **完璧を求めない。** ユーザーが「それで良いです」「おまかせ」と言ったら、こちらの推奨案で確定して先に進む。1つのテーマで議論が3往復を超えたら、暫定案を提示して次へ進むことを提案する。
- **各テーマの終わりに合意内容を1〜3行で要約**してから次のテーマに移る。
- 壁打ち全体をこのセッション内で完結させる。文脈が途切れると議論が巻き戻るため、途中で別作業を挟まない。

### 壁打ちで決めること(この順に進める)

READMEの原典どおり、以下の6テーマを順番に扱う:

| # | テーマ | 質問例 |
|---|--------|--------|
| 1 | **スコープ** | MVP?フル機能?ポートフォリオ用途? |
| 2 | **ユーザーの種類** | ロールは何種類?権限の違いは? |
| 3 | **コアフロー** | ユーザーが最も頻繁に行う操作は何? |
| 4 | **画面一覧** | 何画面必要?各画面の目的は? |
| 5 | **画面遷移** | 画面間の導線は? |
| 6 | **データ構造** | 主要なエンティティは?関係性は? |

各テーマの進め方:

1. **スコープ** — 最初に確定する。以降のすべての判断基準になる。「MVP(コア機能のみ)/フル機能/ポートフォリオ(見栄え重視)」等の選択肢で聞く。
2. **ユーザーの種類** — コンセプトから想定されるロール案を提示して選ばせる(例: 「一般ユーザーのみ/一般+管理者/ゲスト+会員+管理者」)。ロールごとに権限の違いを1行で定義する。
3. **コアフロー** — 「このアプリで最も重要な1本の体験」を特定する。候補を2〜3案提示して選ばせ、選ばれたフローをステップ列(例: `一覧を見る → 詳細を開く → アクション実行 → 結果確認`)で書き下して確認する。
4. **画面一覧** — コアフローとロールから必要な画面を洗い出し、たたき台の一覧(名称+目的)を提示する。「これで足りるか/削る画面はあるか」を選択式で確認する。スコープがMVPなら画面数を絞る方向に誘導する。
5. **画面遷移** — 画面一覧をもとに導線をテキストで提示し(`ログイン → ダッシュボード → …`)、違和感がないか確認する。分岐(ロール別の入口、認証要否)も押さえる。
6. **データ構造** — 画面一覧から逆算した主要エンティティと関係性(1対多など)のたたき台を提示して確認する。カラム定義まで詰めない。詳細はPhase 2で固める。

### たたき台を先に出す

各テーマでは、ゼロから質問するのではなく**コンセプトから推測したたたき台を先に提示し、選択・修正してもらう**形にする。ユーザーの認知負荷を下げ、議論を速く収束させるため。

### 合意のファイル化

6テーマの合意が揃ったら、全体を一度サマリとして提示してユーザーの最終確認を取り、`docs/brainstorm-notes.md` に書き出す。
**後工程に渡すのは常にファイル。口頭(チャット内)の合意で終わらせない。** 次フェーズはこのファイルだけを入力として動けなければならない。

## 成果物

`docs/brainstorm-notes.md`(`docs/` が無ければ作成する)。構成:

```markdown
# 壁打ちメモ(Phase 1 成果物)

## スコープ
<MVP/フル機能等の決定と、その判断理由を1〜2行>

## ロール定義
| ロール | 説明 | 権限 |
|--------|------|------|

## コアフロー
<最も重要なユーザー体験の流れをステップ列で>

## 画面一覧
| 画面名 | 目的 | 簡易説明 | 対象ロール | 認証 |
|--------|------|----------|-----------|------|

## 画面遷移の全体像
<テキストまたはMermaidで導線を記述。分岐条件も明記>

## データ構造(概要)
<主要エンティティと関係性。カラム定義はPhase 2で詰める>

## 保留事項・未決定事項
<壁打ちで先送りにした論点があれば列挙>
```

書き出したらパスをユーザーに伝え、内容の修正希望がないか一度だけ確認する。

## 次のフェーズ

Phase 2(設計仕様書)へ進む。ユーザーに次のように案内する:

- 「次は `/draft:design-spec` で、この壁打ち結果を設計仕様書に落とし込みます。」
- 新しいセッションで開始することを推奨する(コンテキストを軽く保つため)。`docs/concept.md` と `docs/brainstorm-notes.md` がファイルとして残っているので、新セッションでも文脈は引き継がれる。
- ただしユーザーがこのまま同一セッションで続けたい場合はそれに従う。

## 注意

- **フェーズを飛ばさない。** コンセプトなしで壁打ちを始めない(方向性がブレて手戻りが大きくなる)。同様に、壁打ちを飛ばして設計仕様書に進めたがるユーザーには、最低限の6テーマだけでも確認することを勧める。
- **壁打ちは1セッション(1チャット)に収める。** 文脈が途切れると議論が巻き戻る。
- **選択式で聞き、直感で選ばせる。** 後で変えられるので、ユーザーが迷っていたら推奨案を示して先に進める。
- **「それで良いです」で先に進めるのが大事。** 完璧を求めると止まる。未決定事項は「保留事項」としてファイルに残せば十分。
- **このフェーズで実装・技術選定に踏み込まない。** 技術スタックの確定はPhase 3、詳細なデータモデル(カラム定義・ER図)はPhase 2の仕事。話題が出たらメモに留めて先へ進む。
- **成果物は必ずファイルに書き出す。** チャット内の合意のまま終えると、次フェーズが古い・欠けた情報で設計してしまう。
