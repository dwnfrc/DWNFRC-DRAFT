# DRAFT 手動フロー

DRAFTの各フェーズを、Claude.ai のチャットにプロンプトを貼りながら手で回すための手順集。
Phase 0（コンセプト）は人間が `concept.md` をメモ書きレベルで用意する。Phase 1〜4は Claude.ai、Phase 5は Claude Code のチャットで進める。

---

## Phase 1: 壁打ち

Claude.ai の1チャットで対話的に進める。concept.mdを添付して開始。

### プロンプト

```
添付のコンセプトをWebアプリケーション化したいです。
どのような画面が必要でどのようなフローになるか壁打ちをさせてください。
```

### 進め方のコツ

- **1チャットに収める。** 壁打ちの文脈が途切れると議論が巻き戻る
- **選択式で聞き返してくれる**ので、直感で選んでいい。後で変えられる
- **「それで良いです」と言って先に進める**のが大事。完璧を求めると止まる
- 最後に合意事項を `brainstorm-notes.md` として書き出させ、`docs/` に保存する

---

## Phase 2: 設計仕様書

Phase 1と同じチャット、または `brainstorm-notes.md` を新しいチャットに渡す。

### プロンプト

```
ここまでの議論を設計仕様書としてMarkdownファイルにまとめてください。

含める内容:
1. プロダクト概要（コンセプト、方針（対応言語含む）、UI設計原則）
2. ユーザーロールとフロー
3. 画面一覧（全画面の名称・目的・レイアウト・認証要否）
4. コア画面の詳細仕様（各ステップで何が表示され、何が操作できるか。
   エラー・空・ローディング状態も含む）
5. データモデル（ER図 + 各テーブルのカラム定義）
6. 画面遷移図（Mermaid形式で別ファイル）
```

### 深掘りが必要な場合

```
[特定の画面名]の詳細仕様を詰めたいです。
- ユーザーが画面を開いたとき何が見えるか
- 操作可能なアクションは何か
- 各アクション後の状態遷移は何か
- エラー時・データが空のとき何が見えるか
- レイアウト（左右分割？中央集中？グリッド？）
```

```
UIコンポーネントのレイアウトパターンを体系化したいです。
画面の目的別に最適なレイアウトを定義し、全画面にマッピングしてください。
```

## Phase 3: 開発ドキュメント

新しいチャットを開始。設計仕様書 + PROJECT-PLAYBOOKテンプレート + 技術スタック参考資料（あれば）を添付。

### プロンプト

```
添付の設計仕様書とドキュメントテンプレート（PROJECT-PLAYBOOK）を元に、
開発に着手できるようドキュメント一式を作成してください。

テンプレートの構成に従い、以下のドキュメントを埋めてください:
1. PRD（01_prd.md）
2. System Design Doc（02-01_system-design-doc.md）
3. Feature Design Doc（02-02_feature-design-doc.md）— テンプレートのまま残す
4. Dev Setup（03_dev-setup.md）
5. Deployment Procedure（04_deployment-procedure.md）
6. Operation Runbook（05_operation-runbook.md）

あわせて以下も作成してください:
- docs/README.md（ドキュメントインデックス）: 各ドキュメントへのリンク + 内容の
  一行説明（設計仕様書・画面遷移図も含む）、読み順ガイド。実在するファイルのみ載せる
- .github/PULL_REQUEST_TEMPLATE.md（PRテンプレート）: 概要 / 関連ドキュメント /
  変更内容 / 動作確認 / チェックリスト。promotion PR（リリース）の場合に書くことも案内

技術スタックの前提:
- フロントエンド: [Next.js / React / etc.]
- バックエンド: [NestJS / Express / etc.]
- DB: [PostgreSQL / MySQL / etc.]
- 認証: [NextAuth / Firebase Auth / etc.]
- ホスティング: [Cloud Run / Vercel / etc.]
- IaC: [Terraform / なし]
```

### 技術スタックが未定の場合

```
技術スタックをまだ決めていません。
ポートフォリオ用途で、[フロントはReact系、バックはTypeScript]という方向性だけあります。
選択肢を提示して一緒に決めたいです。
```

## Phase 4: 実装準備

Phase 3と同じチャットで続けるか、設計仕様書・開発ドキュメント一式・p-4テンプレート2点（CLAUDE.template.md、claude-code-prompts.md）を新しいチャットに渡す。

### プロンプト

```
ここからはClaude Codeで実装を進めようと思っています。
以下の2ファイルを作成してください。

1. CLAUDE.md — Claude Codeが自動で読むプロジェクトのコンテキストファイル
   - templates/p-4/CLAUDE.template.md の構成に従う
   - Tech Stack、ディレクトリ構成、設計方針、コマンド一覧、docsへの参照パス、実装の規律

2. claude-code-prompts.md — Claude Codeに1つずつ投げるステップ別プロンプト集
   - templates/p-4/claude-code-prompts.md の構成に従う
   - 各ステップに明確なゴール（「〜が動く状態」）
   - 依存関係順（骨格 → DB → API → 認証 → UI → インフラ → テスト）
   - 1ステップ = Claude Codeの1チャットで完結する粒度
```

## Phase 5: 実装

```
リポジトリ初期化
├── CLAUDE.md を配置
├── docs/ にPhase 2〜4の成果物を配置（claude-code-prompts.md 含む）
└── Claude Code を開く
    ├── Chat 1: Step 1（〜2）をコピペ → 動作確認
    ├── Chat 2: Step 3 をコピペ → 動作確認
    ├── Chat 3: Step 4 をコピペ → 動作確認
    ├── ...
    └── Chat N: 最終Step → 動作確認
```

### チャット分割の目安

| チャット | Steps | 目安ファイル数 |
|----------|-------|-------------|
| Chat 1 | 骨格 + DB | 〜20ファイル |
| Chat 2 | API | 〜30ファイル |
| Chat 3 | 認証 | 〜10ファイル |
| Chat 4 | コアUI | 〜20ファイル |
| Chat 5 | 残りUI | 〜20ファイル |
| Chat 6 | インフラ + テスト | 〜20ファイル |

### 問題が起きたら

```
[エラーメッセージをコピペ]

上記のエラーが出ています。場当たり的に直さず、根本原因を特定した上で、
原因と修正方針を説明してから修正してください。
```

### 進め方のコツ

- **新しいチャットではCLAUDE.mdが自動で読まれる**ので、プロジェクト説明は不要
- **前のチャットで作ったファイルはディスクに残っている**ので、引き継ぎ説明も不要

## イテレーション: 機能追加

### 壁打ちの始め方

Claude.aiで新しいチャットを開き、設計仕様書・System Design Doc・PRDを添付して開始する。

```
添付のドキュメントは既存プロジェクトの設計仕様書・System Design Doc・PRDです。

このプロジェクトに以下の機能を追加したいです:
{追加したい機能の概要（2〜3行で十分）}

docs/02-02_feature-design-doc.md のテンプレートに従って、
Feature Design Docを作成するための壁打ちをさせてください。
最終的な成果物は docs/features/ に配置します。

特に以下を一緒に検討したいです:
- この機能はPRDのKPI・スコープに照らして作る価値があるか
- 既存の画面・APIへの影響範囲
- DBスキーマの変更が必要か
- 新規画面・エンドポイントが必要か
```

### Feature Design Docの成果物化

壁打ちが終わったら、同じチャットで成果物化する:

```
ここまでの議論をFeature Design Docとしてまとめてください。
docs/02-02_feature-design-doc.md のテンプレート構成に従ってください。
ファイル名は YYYYMMDD-HHMM_{機能名}.md としてください。
HHMMは丸めずに現在の正確な時刻を使ってください（例: 14:46なら 20260308-1446_notification.md）。
配置先は docs/features/ です。
```

### 実装プロンプトの生成

Feature Design Docができたら、同じチャットで続けて実装プロンプトを生成する:

```
このFeature Design Docの内容を、Claude Codeに投げる実装プロンプトに変換してください。

以下の原則に従ってください:
- 実装の依存関係順に並べる（スキーマ変更 → API → UI）
- 各ステップに明確なゴール（「〜が動く状態」）を定義
- 参照すべきdocsファイルを明示
- 変更規模が大きい場合はチャットを分割する粒度で

出力形式: コードブロック内にプロンプトを記載（そのままClaude Codeにコピペできる形）
(mdのフォーマットが崩れて1クリックでコピーできないことがあるので注意してください。)
```

### Claude Codeでの実行

生成されたプロンプトをClaude Codeの新しいチャットにコピペして実行する。

Feature Design Doc自体は `docs/features/` に配置済みなので、
プロンプト内で `docs/features/YYYYMMDD-HHMM_{機能名}.md を参照` と指示すればClaude Codeが読みに行ける。

### ソースDoc更新の組み込み

実装プロンプトの末尾に以下を含めておけば、Claude Codeが実装と同じチャット内で更新してくれる:

```
実装完了後、Feature Design Docの変更内容に基づいて、
ソース層のドキュメントを現在の実装に合わせて更新してください。
更新対象は事実の所有権マップに従います:
- 画面・UX・レイアウト・認証要否の変更 → design-spec.md（+ screen_flow.mermaid）
- API・データモデル・ルーティング・技術選定の変更 → 02-01_system-design-doc.md
- 主要機能（ユーザーストーリー）・KPI・スコープの変化 → 01_prd.md
- 環境構築・コマンド・デプロイ・運用の変化 → 03〜05

更新は「現在の状態」の書き換えとして行い、更新履歴・編集ログ・
Feature Design Docへの参照注記はドキュメントに残さないでください。
```

### コミット前の独立レビュー

新しいチャット（経緯を知らないコンテキスト）に、実装のdiff・FDD・更新後のドキュメントだけを渡してレビューさせ、指摘を反映してからコミットする。

