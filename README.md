# DRAFT - Document-driven, Reproducible, AI-powered, Full-stack Toolkit

AI駆動のフルスタック開発フレームワーク。
対話で設計を練り上げ、コーディングエージェントで実装する。再現可能な手順で、誰でも同じ品質のアウトプットを出せる。

**DRAFT** の名前には2つの意味がある:

- **設計の下書き（Draft）** — いきなりコードを書かない。コンセプトから設計書を段階的に練り上げ、確かな設計図ができてから実装に入る
- **レースのドラフティング（Drafting）** — 前を走る車の空力を借りて加速する技術。AIという先行者の力を借りて、開発を加速する

---

## 全体像

```
Phase 0    Phase 1      Phase 2       Phase 3        Phase 4        Phase 5
コンセプト → 壁打ち →  設計仕様書 →  開発ドキュメント →  実装準備 →    実装
(人間)     対話型       成果物型       成果物型        成果物型       コード生成

                                        ┌──────────────────────────────────────────────┐
                                        │  イテレーション（機能追加）                     │
Phase 5完了後 →                          │  壁打ち → Feature Design Doc                 │
                                        │  → 実装プロンプト生成 → 実装                  │
                                        │  → ソースDoc更新（所有権マップに従う）          │
                                        └──────────┬───────────────────────────────────┘
                                                   └→ 繰り返し
```

**原則: 後工程に渡すのは常にファイル。口頭の合意ではなくMarkdownに書き出す。**
**原則: レビューは制作した本人と別の、経緯を知らない新鮮なコンテキストで行う。**（サブエージェントや新しいセッションに成果物だけを渡す）
**原則: ドキュメントがソース、実装は成果物。実装は常にソース層のドキュメントから再現できる状態を保つ。**

---

## ドキュメント体系（3層と所有権）

DRAFTの根本思想: **ドキュメントがソースであり、実装はそこから生成される成果物。**
この思想を成立させるため、ドキュメントを3層に分けて管理する。

### 3層構造

| 層 | ファイル | 役割 |
|----|---------|------|
| **ソース** | `design-spec.md`, `screen_flow.mermaid`, `01_prd.md`, `02-01_system-design-doc.md`, `03_dev-setup.md`, `04_deployment-procedure.md`, `05_operation-runbook.md` | 実装を再現するための正。常に「現在の状態」を反映する |
| **派生** | `CLAUDE.md`, `claude-code-prompts.md`, `docs/README.md` | ソースから再生成可能。今のAIの制約（コンテキスト分割等）への足場 |
| **アーカイブ** | `concept.md`, `brainstorm-notes.md`, `features/`（FDD） | 経緯の記録。再現には使わない。書き換えず蓄積する |

- 実装を変えたら**ソース層**を必ず現在の状態に更新する。更新履歴は残さない（経緯はFDDとgitが担う）
- **派生層**は古くなったらソースから作り直せばよい。ソースと食い違ったらソースが正
- **アーカイブ層**は更新しない。作成時点のスナップショットであることが役割

### 事実の所有権

同じ事実を複数のドキュメントに書かない（コードのDRY原則と同じ）。事実ごとに所有ドキュメントを1つ決め、他のドキュメントは転記せず参照する:

| 事実 | 所有ドキュメント |
|------|----------------|
| なぜ作るか・誰のため・主要機能（ユーザーストーリー）・KPI・スコープ外 | `01_prd.md` |
| 画面の存在・目的・レイアウト・認証要否・UXの振る舞い・コアフロー | `design-spec.md` + `screen_flow.mermaid` |
| 技術選定（ADR）・アーキテクチャ・API設計・データモデル・ルーティング・エラーハンドリング・i18n実装 | `02-01_system-design-doc.md` |
| 環境構築・コマンド体系（makeターゲット名）・ブランチ戦略・リリースフロー | `03_dev-setup.md` |
| 実コマンドの定義 | `Makefile` |
| デプロイ手順・ロールバック | `04_deployment-procedure.md` |
| 監視・障害対応 | `05_operation-runbook.md` |
| 変更の経緯（なぜ変えたか） | `docs/features/`（FDD） |

- **更新の判定はこの表で行う。** 変わった事実の所有ドキュメントを更新する。変更規模の大小は関係ない
- **データモデルの所有権はPhase 3で移る。** Phase 2では `design-spec.md`（論理設計）で固め、Phase 3で `02-01_system-design-doc.md`（ER図+ORMスキーマ）へ引き渡す。以降のスキーマ変更はSystem Design Docのみを更新し、design-specのデータモデル章は設計時ドラフトとして残す（追随させない）
- **System Design Docの画面一覧はルーティング表に留める。** ルート → 画面の対応だけを持ち、画面の目的・レイアウト・認証要否は design-spec を参照する

### 方法論自身への適用

この3層はDRAFTリポジトリ自身にも適用される:

| 層 | 対象 |
|----|------|
| **ソース** | このREADME（方法論のspec）と `templates/`（テンプレート原本） |
| **派生** | `draft-plugin/` のスキル群とテンプレートコピー — READMEの**実装**。方法論のspecで食い違ったらREADMEが正（実行の詳細 — 対話手順・ツール使用 — はスキルが固有に所有する）。Claude Codeプラグインとして配布される |
| **アーカイブ** | 変更の経緯（git履歴） |

ルール（決まり事）にも層がある。意図・検証に関わるルールは**恒久**、今のAIの制約（コンテキスト分割等）を補うルールは**足場**であり、足場は理由が失効したら削除する。方法論は足すだけでなく、削って新陳代謝させる（このリポジトリでは evolve スキルのルールGCが担う）。

## Phase 0: コンセプト（人間が用意する）

### ゴール

「何を作りたいか」を1ページで言語化する。完璧でなくていい。

### 使用テンプレート

`templates/p-0/concept.md`

### やること

- アイデアをメモ書きレベルで書く
- 以下が含まれていると後工程がスムーズ:
  - 誰のためのプロダクトか
  - どんな体験を提供するか
  - 自分がなぜ作りたいか（ポートフォリオ？事業？学習？）

### 成果物

`concept.md`

### Tips

- 構成が雑でも問題ない。Phase 1でAIが構造化してくれる
- 「コンセプトに自信がない」状態でもPhase 1に進んでいい。壁打ちで磨かれる

## Phase 1: 壁打ち

### ゴール

コンセプトを「画面とフローが見える」レベルまで具体化する。`concept.md` を入力に、1つの対話セッションで進める。

### 壁打ちで決めること

| 項目 | 質問例 |
|------|--------|
| **スコープ** | MVP？フル機能？ポートフォリオ？ |
| **ユーザーの種類** | ロールは何種類？権限の違いは？ |
| **コアフロー** | ユーザーが最も頻繁に行う操作は何？ |
| **画面一覧** | 何画面必要？各画面の目的は？ |
| **画面遷移** | 画面間の導線は？ |
| **データ構造** | 主要なエンティティは？関係性は？ |

### 成果物

`brainstorm-notes.md` — 以下の合意事項をMarkdownに書き出したもの。セッション内の合意のままにせず、必ずファイルに書き出して `docs/` に置く（原則「後工程に渡すのは常にファイル」）:

- 画面一覧（名称 + 目的 + 簡易説明）
- 画面遷移の全体像
- ロール定義
- コアフロー（最も重要なユーザー体験の流れ）

### Tips

- **選択式で聞き、直感で選んで先に進める。** 決定は後で変えられる。完璧を求めると止まる
- **1セッションに収める。** 壁打ちの文脈が途切れると議論が巻き戻る

## Phase 2: 設計仕様書

### ゴール

Phase 1の合意を「設計仕様書」として書き出す。開発者（自分含む）が読んで画面が作れるレベル。
`brainstorm-notes.md` を入力にする。

### 使用テンプレート

`templates/p-2/design-spec.md`

### 設計仕様書に含めること

1. プロダクト概要（コンセプト、方針（対応言語含む）、UI設計原則）
2. ユーザーロールとフロー
3. 画面一覧（全画面の名称・目的・レイアウト・認証要否）
4. コア画面の詳細仕様（各ステップで何が表示され、何が操作できるか。エラー・空・ローディング状態も含む）
5. データモデル（ER図 + 各テーブルのカラム定義）
6. 画面遷移図（Mermaid形式で別ファイル）

コアフローに関わる画面や仕様が曖昧な画面は、書きっぱなしにせず深掘りして詰める（開いたとき何が見えるか・操作できるアクション・各アクション後の状態遷移・エラー/空のときの表示・レイアウトパターン）。

### 成果物

| ファイル | 内容 |
|----------|------|
| `design-spec.md` | プロダクト設計の全体像 |
| `screen_flow.mermaid` | 画面遷移図 |

### Tips

- **仕様書は「長くていい」。** Phase 3以降でAIが参照するための情報源
- **データモデルはこの段階で固める。** 後から変えるとAPI設計から全部やり直し。所有権はPhase 3でSystem Design Doc（ER図+ORMスキーマ）へ移り、以降のスキーマ変更はSystem Design Docのみを更新する
- **エラー・空・ローディング状態も仕様化する。** 正常系だけの仕様書だと、実装時に状態設計が抜ける
- **i18nの要否はこの段階で決める。** プロダクト方針に対応言語を明記する（不要なら「日本語のみ」）。後付けは高くつく
- **Mermaid図は別ファイルにする。** 仕様書本文に埋め込むとAIが図を読み飛ばすことがある

## Phase 3: 開発ドキュメント

### ゴール

設計仕様書を「開発着手できる」レベルの技術ドキュメントに変換する。

### 使用テンプレート

PROJECT-PLAYBOOK のドキュメントテンプレート一式:

| テンプレート | 内容 |
|-------------|------|
| `01_prd.md` | 概要/背景と目的、ターゲットユーザーと課題、主要機能（ユーザーストーリー）、成功指標（KPI）、スコープ外 |
| `02-01_system-design-doc.md` | Goal/Non-Goal、アーキテクチャ概要、技術選定(ADR)、ルーティング、API設計、データモデル、セキュリティ・パフォーマンス、エラーハンドリング、i18n、テスト戦略、モニタリング・ログ |
| `02-02_feature-design-doc.md` | 背景と目的、Goal/Non-Goal、影響範囲、実装アプローチ、詳細設計、テスト方針、ロールアウト計画 |
| `03_dev-setup.md` | 環境構築手順、必要なツール・アカウント、ローカル開発サーバー起動、テスト実行、ブランチ戦略・リリースフロー、トラブルシューティング |
| `04_deployment-procedure.md` | 各環境へのデプロイ手順、リリース前チェックリスト、ロールバック手順、デプロイ後確認、緊急時連絡先 |
| `05_operation-runbook.md` | モニタリング・ログ設計、監視ポイントとアラート、よくある障害と対処法、ログ確認方法、定期メンテナンス |

### 進め方

設計仕様書とPROJECT-PLAYBOOKテンプレートを入力に、6ドキュメントを埋める。技術スタックが未定なら、用途と方向性（例: フロントはReact系、バックはTypeScript）を前提に選択肢を比較して先に確定させる。
Feature Design Doc（02-02）はテンプレートのまま `docs/` に残す（イテレーションで雛形として使う）。

### 成果物

PROJECT-PLAYBOOKテンプレートに従った6ファイル + `docs/README.md`（ドキュメントインデックス）+ `.github/PULL_REQUEST_TEMPLATE.md`（PRテンプレート）。

### Tips

- **技術選定はこのフェーズで確定させる。** Phase 4以降で変えると手戻りが大きい
- **コマンドは標準makeターゲットに統一する。** 実コマンド（pnpm等）はMakefileに集約し、docsとCLAUDE.mdは `make dev` / `make test` 等のターゲット名だけを参照する。どのプロジェクトでも同じ操作体系になり、コマンド変更時の修正はMakefile1箇所で済む（Windowsはmake非標準のためWSL前提）
- **ADR（Architectural Decision Records）は必ず書く。** 「なぜこの技術を選んだか」がポートフォリオの差別化になる
- **セキュリティ章を必ず埋める。** 特に認可の権限マトリクス（ロール×リソース）。実装レビューで認可漏れを検出するには、照合先となる正がSystem Design Docに必要
- **API設計はリクエスト/レスポンスのボディまで書く。** バックエンド実装時のプロンプトで「API設計に従って」と指示できる
- **DBスキーマは実際のコード（Prisma/Drizzle等）で書く。** そのままコピペで使える状態にする

## Phase 4: 実装準備

### ゴール

実装フェーズに渡す2つのファイルを作る: コンテキストファイル（CLAUDE.md）とステップ別プロンプト集（claude-code-prompts.md）。
前提知識（毎セッション読まれるCLAUDE.md）と作業指示（都度渡すprompts.md）の分離がこのフェーズの核心。

### 使用テンプレート

| テンプレート | 用途 |
|-------------|------|
| `templates/p-4/CLAUDE.template.md` | Claude Codeが自動読み込みするプロジェクトコンテキスト |
| `templates/p-4/claude-code-prompts.md` | ステップ別プロンプト集の構成ガイド |

### CLAUDE.md に書くこと / 書かないこと

| 書く | 書かない |
|------|---------|
| 確定した技術スタック | ステップ別の作業指示 |
| ディレクトリ構成 | 変わりやすい情報（TODO、進捗） |
| 重要な設計方針（3〜5個） | 冗長な説明 |
| 開発コマンド一覧（標準makeターゲット） | 生コマンドの複製（真実源はMakefile） |
| docsへのファイルパス | |
| コーディング規約 | |
| 実装の規律（場当たり修正・TODOコメントの禁止など） | |

### ステッププロンプトの設計原則

| 原則 | 理由 |
|------|------|
| **1ステップ = 1チャット** | コンテキスト劣化（自動要約による詳細の損失・注意の希釈）の防止 |
| **ゴールを「動作確認可能な状態」で定義** | Claude Codeが完了判断できる |
| **参照すべきdocsファイルを明示** | 必要な情報だけ読ませる |
| **「まだやらないこと」を明示** | スコープクリープ防止 |
| **依存関係順に並べる** | 前のステップの成果物が次の前提になる |

### 推奨ステップ構成（Webアプリの場合）

| Step | 内容 | 粒度の目安 |
|------|------|-----------|
| 1 | モノレポ骨格 + 開発環境 | フレームワーク初期化、Docker、.env、Makefile |
| 2 | DBスキーマ + シード | テーブル定義、デモデータ |
| 3 | バックエンドAPI（CRUD） | 全エンドポイント、バリデーション |
| 4 | 認証 | OAuth、ガード、protected routes |
| 5 | 共通UIコンポーネント | レイアウト、デザインシステム |
| 6 | コア画面 | 最も重要な画面 |
| 7 | 残り画面 | 2nd priority の画面群 |
| 8 | インフラ + CI/CD | Terraform、GitHub Actions、Dockerfile |
| 9 | テスト + 仕上げ | Unit、E2E、Lint、KPI計測の導入 |

### 成果物

| ファイル | 配置場所 | 用途 |
|----------|----------|------|
| `CLAUDE.md` | リポジトリルート | Claude Codeが自動読み込み |
| `claude-code-prompts.md` | `docs/` | ステップ別プロンプト（派生層。docsと食い違ったらdocsが正） |

### Tips

- **Step 1〜2は1セッションにまとめてもOK。** 骨格 + スキーマは依存が近い
- **UI実装ステップが最も重い。** コア画面と残り画面は分ける
- **インフラは後回しで良い。** ローカルで動くものを先に作る

## Phase 5: 実装

### ゴール

プロンプト集を順に実行し、動くアプリケーションを完成させる。

### 進め方

リポジトリを初期化し、CLAUDE.mdをルートに、Phase 2〜4の成果物（claude-code-prompts.md 含む）を `docs/` に配置する。
`docs/claude-code-prompts.md` のステップを**1つずつ新しいセッションで**実行し、動作確認をパスしてから次のステップへ進む（同一セッションでの実行をプロンプト集が明記しているステップ群のみ例外）。

### ステップ間の動作確認

各ステップの最後に必ず動作確認する:

| Step | 確認方法 |
|------|----------|
| 骨格 | `make dev` で両方のサーバーが起動する |
| DB | `make db-push` + `make db-seed` + `make db-studio` でデータが見える |
| API | `curl` で各エンドポイントが200を返す |
| 認証 | ブラウザでOAuthログインが通る |
| UI | ブラウザでコアフローが一通り動く |
| インフラ | `terraform plan` がエラーなく通る |
| テスト | `make test` が全パス |

### Tips

- **エラーは場当たり的に直さない。** 根本原因を特定し、原因と修正方針を説明してから修正する。症状だけ抑えるパッチを重ねると後のステップで修正困難になる
- **1ステップが大きすぎると感じたら分割する。** 完璧な粒度は事前にはわからない
- **実装がdocsから逸脱したら、その場でソース層を更新する。** 合意して行った設計変更（API形状・スキーマ・UX調整）は、事実の所有権マップに従い所有ドキュメントへ反映してからステップ完了とする。Phase 5完了直後の最初の壁打ちが古いドキュメントを入力にしないための規律
- **生成されたコードは必ず独立レビューにかける。** 経緯を知らない新しいコンテキスト（サブエージェント/新セッション）にdiffだけを渡す。特にセキュリティ（認証ガード、バリデーション）、残骸（TODO/FIXMEコメント、実ファイル追加済みディレクトリの `.gitkeep`）、docs整合（実装が設計ドキュメントから無断で逸脱していないか）

## イテレーション: 機能追加サイクル

Phase 5完了後、機能追加は以下のサイクルで回す。
Phase 0〜5のフル構築に比べて軽量で、1機能あたり壁打ち〜実装まで短時間で完結する。

### フロー

```
機能の着想（ユーザーの声・KPIの計測値も着想の入力になる）
    ↓
壁打ち
  入力: 設計仕様書 + System Design Doc + PRD + 機能の概要
  出力: Feature Design Doc
    ↓
実装プロンプト生成（同じセッション）
  入力: Feature Design Doc
  出力: 実装セッションに投げるプロンプト
    ↓
実装
  入力: CLAUDE.md + 生成したプロンプト
  1セッションで完結する粒度
    ↓
ソースDoc更新（実装と同じセッション）
  実装完了後、変更内容を所有権マップに従いソース層に反映
```

設計（壁打ち〜Feature Design Doc）と実装はセッションを分けてよい。
受け渡すのはFDDファイルで、実装セッションはFDDから実装プロンプトを組み立て直せるため、
生成済みプロンプトが手元に残っていなくても再開できる。

### 壁打ちの始め方

新しいセッションで、設計仕様書・System Design Doc・PRDと機能の概要（2〜3行で十分）を入力に壁打ちする。
`docs/02-02_feature-design-doc.md` のテンプレートに従い、Feature Design Docを作ることをゴールにする。

壁打ちで必ず検討すること:

- この機能はPRDのKPI・スコープに照らして作る価値があるか
- 既存の画面・APIへの影響範囲
- DBスキーマの変更が必要か
- 新規画面・エンドポイントが必要か

#### 変更の大きさで分岐する

| 変更規模 | 判断基準 | 進め方 |
|---------|---------|--------|
| **小** | 既存画面内の修正、APIパラメータ追加程度 | 壁打ちなしでFeature Design Docを直接書いてもOK |
| **中** | 新規画面1〜2枚追加、スキーマ変更あり | 壁打ちでFeature Design Docを作成 → 1セッションで実装 |
| **大** | 新ロール追加、コアフロー変更 | 壁打ちでFeature Design Doc作成 → 複数セッションで実装 → ソースDoc更新 |

規模はプロセス（壁打ちの要否・セッション分割）の判断にのみ使う。**どのドキュメントを更新するかは規模ではなく所有権マップで決まる**（例: 新規画面が入る中規模でも design-spec.md は更新する）。

### Feature Design Docの成果物化

壁打ちが終わったら、同じセッションで成果物化する。`docs/02-02_feature-design-doc.md` のテンプレート構成に従い、
`docs/features/YYYYMMDD-HHMM_{機能名}.md` に配置する。タイムスタンプは実際の現在時刻を使う（丸めない・推測しない）。

### 実装プロンプトの生成

Feature Design Docができたら、同じセッションで続けて実装プロンプトを生成する。
これはPhase 4でclaude-code-prompts.mdを作ったのと同じ考え方で、
「何を作るか」（Feature Design Doc）と「どう指示するか」（プロンプト）を分離する。

実装プロンプトの原則:

- 実装の依存関係順に並べる（スキーマ変更 → API → UI）
- 各ステップに明確なゴール（「〜が動く状態」）を定義する
- 参照すべきdocsファイル（最低限、対象FDD）を明示する
- 変更規模が大きい場合はセッションを分割する粒度で書く
- 末尾にソースDoc更新の指示を含める（次節）

### ソースドキュメントの更新

実装完了・動作確認後に、ソース層のドキュメントを現在の実装に合わせて更新する。
ソース層は「現在の状態」を表すドキュメントなので、実装前ではなく実装後に更新する。
実装中に方針が変わることがあるため、確定した状態を反映するのが正しいタイミング。
更新対象は事実の所有権マップに従い、更新は「現在の状態」の書き換えとして行う（更新履歴・編集ログ・FDDへの参照注記は残さない）。

この更新指示は実装プロンプトの末尾に組み込んでおく。別作業にすると忘れるが、プロンプトに含めれば実装と同じセッション内で実行される。

コミット前に、経緯を知らない新しいコンテキスト（サブエージェント/新セッション）に
diff・FDD・更新後ドキュメントだけを渡し、実装とドキュメントをレビューさせる。

**なぜ実装後か:**

- ソース層のドキュメントは次のイテレーションの壁打ちで入力として使う
- 常に「現在の状態」を反映していないと、AIが古い情報で設計してしまう
- Feature Design Docは「変更の経緯」（なぜ変えたか）、ソース層は「現在の状態」（今どうなっているか）という役割分担
- この役割分担のため、更新履歴・編集ログ・FDDへの参照注記を正本ドキュメントに残さない（経緯はFDDとgit履歴が担う）。例外はADRで、設計判断が変わった場合は「現在の設計のなぜ」としてADRを追加・更新する

### Branch/Commitの作成

ブランチ運用は `docs/03_dev-setup.md` のブランチ戦略に従う。未定義ならこの時点で決めてdev-setup.mdに追記する。
小規模な変更や構築初期はmain直コミットでよく、スキーマ変更・新規画面など影響範囲が広い変更は実装前にブランチを切る。
リリース（本番反映）はブランチではなく、同ドキュメントのリリースフロー（環境ごとのバージョン宣言ファイルを更新するpromotion PR。ロールバック = revert）に従う。
コミットメッセージはリポジトリの流儀に合わせる（既存のコミットを参考にする場合はonelineではなく全文を確認する）。

### Feature Design Docの管理

ファイル名に `YYYYMMDD-HHMM_` プレフィックスをつけ、時系列を明示する。

```
docs/
├── 設計仕様書.md
├── 01_prd.md
├── 02-01_system-design-doc.md
├── 02-02_feature-design-doc.md        ← テンプレート（空のまま残す）
├── features/                           ← 実際のFeature Design Docs（時系列順）
│   ├── 20260310-0918_notification.md
│   ├── 20260310-1446_email-digest.md
│   ├── 20260318-1103_social-ranking.md
│   └── 20260401-1427_ai-scoring.md
├── 03_dev-setup.md
├── 04_deployment-procedure.md
└── 05_operation-runbook.md
```

### Tips

- **壁打ちには必ず設計仕様書・System Design Doc・PRDを入力にする。** 既存の設計との整合性と、機能がKPI・スコープに照らして作る価値があるかを判断させるために必要
- **小さな変更でも Feature Design Doc を残す。** タイムスタンプ付きで蓄積すれば変更履歴になる
- **Feature Design Docをそのまま実装セッションに渡さない。** 設計書と実装指示は役割が違う。実装プロンプトを生成するステップを挟む
- **スキーマ変更がある場合はマイグレーション計画を先に詰める。** 実装後に「マイグレーション通らない」となると手戻りが大きい
- **ソースDocの更新は実装プロンプトに組み込む。** 別作業にすると忘れる。プロンプトに含めれば実装と同じセッション内で自動で実行される

## アンチパターン

| やりがちなこと | なぜダメか | 代わりにやること |
|--------------|-----------|---------------|
| コンセプトなしでいきなり「アプリ作って」 | 方向性がブレて手戻り大量 | Phase 0〜4を踏む |
| 設計仕様書なしで開発ドキュメントを作る | API設計とUI設計が噛み合わない | Phase 2で画面仕様を固めてからPhase 3 |
| 全ステップを1つのセッションで実行 | コンテキストの肥大で品質低下（自動要約による詳細の損失・注意の希釈） | ステップごとにセッションを分ける |
| CLAUDE.mdにステップ別指示を書く | 毎回読まれるので情報過多 | CLAUDE.md = 前提知識、prompts.md = 指示 |
| 技術スタックを実装フェーズで変更 | ドキュメント全体の書き直し | Phase 3で確定させる |
| 動作確認せずに次のステップへ進む | エラーが積み上がって修正困難 | 各ステップ後に必ず動作確認 |
| 機能追加時に設計仕様書を読ませずに壁打ち | 既存設計と矛盾する設計が出てくる | 設計仕様書 + System Design Docを必ず入力にする |
| 機能追加後にソースDocを更新しない | 次の壁打ちで古い情報を元に設計してしまう | 実装プロンプトにドキュメント更新指示（所有権マップ準拠）を組み込む |
| Feature Design Docをそのまま実装セッションに投げる | 設計書には実装に必要な依存順・検証可能なゴール・スコープ境界が無い（役割が違う） | 実装プロンプトを生成してから渡す |
| 同じ事実を複数のドキュメントに書く | 更新漏れで食い違い、どれが正か不明になる | 所有ドキュメントを1つ決め、他は転記せず参照する |
